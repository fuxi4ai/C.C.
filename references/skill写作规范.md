# skill 写作规范 — 三段式骨架 + 硬门禁

> **来源**：[[视频方法论对照与开发流程优化-设计提案]] §3 可借鉴第 2 条 + §4(A)，Doctor 2026-06-13 裁定落 references/。
> **一句话**：skill 不是操作手册。**删「怎么做」，留「什么算好 + 什么红线」**；能用程序判定的事做成自动门禁，别写进散文规则反复念。

---

## §1 · 三段式骨架（每个 SKILL.md 的核心只写三类）

写新 skill / 改旧 skill 时，正文核心收敛到三段：

1. **何时用（When / Triggers）**
   - 触发词、触发场景、适用边界（"什么情况下用"+"什么情况下**不**用"）。
   - Cowork 路由靠 description 命中，触发条件要写在 description 里，准而不泛。

2. **什么算合格（Done / 可验证标准）**
   - 完成的客观标准，**可机器验证**：文件存在于 X 路径 / `Grep "oldname"` 在 dir/ = 0 matches / 脚本输出 = Z。
   - 禁止"做好了""完整""修复了"这类主观陈述（写了等于没写，见 G-X4「可验证=客观」）。

3. **什么红线（Never / 禁止项）**
   - 绝对不能碰的事：环境硬约束、安全禁忌、数据纪律。
   - 红线要短、要硬、要可自查（"看到自己即将 X → 立即停"）。

**怎么做（How）默认删**——把具体步骤留给模型自己组织。**唯一例外**：保留"先打地基再抠细节"这种**骨架级**顺序约束（不是分步说明书），以及确有必要的复杂子流程（如需求问题库这类"什么答案能接受/必须打回"的判定库，可以写厚——因为它是 what 不是 how）。

---

## §2 · 厚薄判别尺（防"轻流程"被误读）

视频口号"写越细越压模型上限"只对一半。判别尺：

| 内容类型 | 厚薄 | 理由 |
|---|---|---|
| **怎么做（how）的分步操作** | **轻**（删，交给模型） | 写死会压模型上限、且易过时 |
| **什么算好（done 标准）** | **厚**（写足，可验证） | done 由模型自己解释 = 完成幻觉根源（G-X4） |
| **什么红线（环境/安全硬约束）** | **厚**（写死，硬门禁） | 不是"压上限的步骤"，是踩过的血坑 |

**必须厚的红线清单（环境硬约束，照抄进相关 skill 的 Never 段）**：
- 🔒 **数据真实性铁律**：宁可缺数不可假数；占位/样本不入库、不进报告（白泽占位活体案）。
- ⛔ **G-X2 沙箱禁 git 写操作**：`add/commit/push/...` 一律贴命令给 Doctor；连只读 `git status` 在 FUSE 盘也留 index.lock。
- ⛔ **G-X7 zip 在 FUSE/挂载盘产 0 字节坏包**：压缩改在 `/tmp` 原生盘，校验后再 cp 回。
- ⛔ **沙箱 sqlite 直写 disk-I/O**：拷出 /tmp 改、整库写回。
- ⚠️ **Cowork 落点规则**：项目文件/全局文件/AI4ME 交付物/临时草稿四分。

判别一句话：**压 how 可以轻，定 what 与守红线必须厚。** 所以不是"把 skill 写薄"，是"删 how、留 what+红线"。

## §3 · 能判定的做硬门禁，不写进散文

凡是**能用程序/Grep/脚本客观判定**的约束，做成**自动门禁**（卡死、报错回退），而不是写一条散文规则靠 CC 自觉反复念——靠自觉会失效（G-X2 再犯即活体）。

现有硬门禁范例（复用，不重造）：
- `validate-frontmatter.py` — frontmatter 合规闸。
- `check-cascade.py` — 改沉淀事实前的级联依赖闸。
- checkpoint / progress 文件 — 长任务进度门禁。
- PRD 三态模板 — checkbox 默认 `[ ]`，CC 只能填 `[?]/[!]/[~]`，看到 `[✓]` 即停（G-X4 硬约束）。

新写 skill 时自问：**这条规则能不能用一行脚本判定？能 → 做门禁；不能 → 才写进 done 标准。**

## §4 · Cowork 环境约束（写 skill 必记）

- **多文件 skill 在 Cowork 退化为单 SKILL.md**：子目录不保留，复杂逻辑要在项目里留完整源码副本作 fallback（通用教训 · O MY HTML）。
- **三层不自动同步（G-X6）**：源 `brain/.skills/<name>/SKILL.md` → `.skill` 包 → 运行时已安装副本，改源不自动传播。改 skill 必走"改源 → 重打包（/tmp）→ Doctor 重装"三步一体，缺一即假同步。
- **description 决定路由**：Cowork 靠 description 命中触发，写法见三段式 §1.1。

---

## §5 · 新建/修改 skill 对照 checklist

- [ ] description 写清"何时用 + 何时不用"，触发词准而不泛
- [ ] done 标准全部可机器验证（无主观陈述）
- [ ] Never 段含所有相关环境硬约束红线
- [ ] how 分步是否已删（保留骨架级顺序即可）
- [ ] 能判定的约束是否已做成门禁（而非散文规则）
- [ ] 多文件结构是否在项目留 fallback 副本
- [ ] 改源后是否走"重打包 + Doctor 重装"三步（G-X6）
