## 存放位置约定

| 类别 | 存放方案 | 适用场景 |
|------|---------|---------|
| **项目专用 key** | `brain/.tools/.env.{用途}` | graphify、未来项目脚本类 |
| **系统级凭证** | macOS Keychain | GitHub PAT、iCloud、邮箱 |
| **多设备共享** | 1Password / Bitwarden（可选） | 跨机器共享、团队协作 |
| **临时调试** | shell session env，**禁止**写入文件 | 一次性测试 |

### 绝对禁止

- ❌ 明文 key 写进 git commit / log / chat 历史 / 任何 markdown
- ❌ key 复制粘贴到对话框、Slack、邮件
- ❌ 共用同一把 key 跨多个项目（粒度按"服务 × 用途"切分）

### E-002 · DashScope API Key 代码硬编码 + 配置明文（2026-05-14 发现）

- **凭证**：清单 #4，DashScope key（用于 DVA Qwen3-ASR/sensevoice 字幕转写）
- **暴露面**：
  1. **代码硬编码**：`Projects/DVA/dyd/dva_asr.py:54` 把完整 key 写成 `os.getenv("DASHSCOPE_KEY", "<完整key>")` 的 default fallback
  2. **配置明文**：`Projects/DVA/dyd/config.yml` 的 `transcript.api_key` 字段
  3. **本次审计对话**：扫描子代理把完整 key 字面值打印进 Cowork 工具结果（一次性，对话结束不可恢复）
- **风险评估**：
  - DVA 项目本身不是 git 仓库 → 未 push 到任何 remote ✓
  - 但任何拷贝 / 同步 / 备份操作都会把 key 一起带走
  - 硬编码 fallback 意味着即使删了环境变量、配置也还能用 → 违反"凭证与代码分离"原则
  - 当前 `transcript.enabled: false`，key 实际未在用
- **整改方案（待 Doctor 决策）**：
  - A. 立即轮换 + 重构代码（删 fallback / config 引用 env）
  - B. 接受风险（如同 E-001，预付费/小余额场景）
  - C. 仅重构代码不轮换（最低成本去硬编码）
- **复盘教训**：
  - `os.getenv("KEY", "完整key")` 这种带 default 的写法是反模式，等同于把 key 写死
  - 配置文件里的 key 字段应该用 `"${ENV_VAR}"` 占位符，由部署时注入

### E-003 · 火山引擎 TOS AK/SK 代码硬编码（2026-05-14 发现）

- **凭证**：清单 #5，`TOS_AK` / `TOS_SK`（DVA 临时音频暂存桶 dragonpalace）
- **暴露面**：
  1. `dva_asr.py:47-48` 两行 `os.getenv("TOS_AK", "<完整AK>")` / `os.getenv("TOS_SK", "<完整SK_base64>")` 硬编码 fallback
  2. 本次审计对话（ephemeral，对话结束不可恢复）
- **风险评估**：
  - DVA 不是 git 仓库 → 未 push 到任何 remote ✓
  - TOS_SK 以 base64 编码，看起来混淆但 base64 ≠ 加密，可秒还原
  - dragonpalace bucket 仅存转写临时音频，数据敏感性低；风险点是 AK/SK 被拿去消耗存储/流量
  - 当前 transcript.enabled: false，TOS 调用路径不活跃
- **整改决策**（Doctor 未明确 · CC 代决 · 2026-05-14）：
  - 采用 **E-002 同款 C 方案**：仅重构代码，不轮换（与 E-002 同等理由）
  - `dva_asr.py` 已去掉两行 fallback；正确存放位置为 `DVA/dyd/.env.dva`
- **状态**：✅ 代码整改完成，等 Doctor 创建 `.env.dva` 填入真实值后完全就位
- **复盘教训**：
  - 与 `os.getenv("KEY", default)` 同款反模式，特别是 TOS_SK 用 base64 这种"伪混淆"更危险——视觉上像乱码，实际一行就能解码
  - 今后所有 AK/SK 类凭证：存 `.env.{用途}`，代码里 `os.getenv("KEY")` 无 default，启动时做 None 检查
