# PRD 交付标准主机制 · 设计提案

> **Doctor 2026-05-20 命题**:"CC 错误自发现率非常高了 · 但偶尔依然出现需要我发现错误才能自觉的现象。偶尔错误难以避免 · 我们要做的是建立一个错误发现机制。我的建议是 · 开发项目时 · 在 PRD 写交付标准 · 完成后打勾 · 所有交付标准完成后(或用户取消后)才算交付成功。"
>
> **本文档** = 设计提案 · 待 Doctor 裁定 · 不是落实方案

---

## §一 · 问题诊断

### 1.1 现象

CC 错误自发现率高(本会话里 CC 自检出"凌晨星级膨胀"/"颗粒度坍缩"/"双线对照过时"等多处问题)· 但**偶尔仍有需要 Doctor 提示才意识到错误**的情况。

### 1.2 今天的真实样本

| 错误 | CC 自检 | Doctor 提示 |
|---|---|---|
| 凌晨 70 位哲学家 ★★★ 膨胀 | ✗ | ✓(Doctor 5/19 20:42 主排序基线介入) |
| 沟通"双线对照" v1.0 应对三主线时仅改表格列 | ✗ | ✓(Doctor 后续 5/20 明确"双线对照迭代为三线对照") |
| brain-save Step 5 在 sandbox 跑 git(违反 5/18 已立规则) | ✗ | ✓(Doctor 提示后才意识到) |
| 朱熹 ★★ 与伊本·西那 ★★★ 不一致 | ✗ | ✓(Doctor 5/20 提示降档) |

CC 自检率 = **0/4** · 全部需 Doctor 发现。这不是错误率问题(本会话 CC 自检率整体很高) · 这是**特定类别错误的盲区**。

### 1.3 盲区机制 = "完成幻觉"

LLM 倾向"积极完成"姿态:写完文件 → 觉得 done。但**done 的定义**完全由 CC 自己解释。三个典型漂移:

- **范围漂移**:任务说"改 brain-save",CC 改完 SKILL.md 觉得 done · 实际"需要打包成 .skill 让 Doctor 重装"也是任务一部分
- **一致性漂移**:任务说"伊本·西那降档",CC 改 4 个文件后觉得 done · 实际西方哲学史 3 处旁证仍写"★★★ 顶级"
- **规则漂移**:任务说"git commit + push",CC 直接 bash 跑 · 忽视 G-X2 "不在 sandbox 跑 git" 规则

**共同根因**:**"done"由完成者自己定义** · 没有外部基准。

## §二 · 现状盘点 + 共同缺陷

| 机制 | 性质 | 致命缺陷 |
|---|---|---|
| **TaskList**(TaskCreate/TaskUpdate) | "做什么"过程任务 | CC 自建 / 自标 completed · 没有"标 completed 应满足什么"的外部基准 · **自评自打勾偏置** |
| **G-X1 第 5 条 self-audit** | 任务结束时强制写"做了什么 + 偏差基线" | 是**事后追溯**不是**事前定义** · CC 写 self-audit 时自己解释自己 · 偏置仍在 |
| **session-log 「遗留问题/待办」** | /save 时回顾本会话 | 事后追溯 · 且只触发于 /save · 中段漏的发现不了 |
| **conch / brain-save / 等 skill 的 Step 流程** | skill 内部标准化步骤 | skill 级别有效 · **项目层级无对应物**(写 brain-save 升级算不算交付了?没有任何标准说"算") |

**共同问题** = 全部是"CC 在 CC 自己定义的标准下评 CC 自己"。

## §三 · 提议机制:三组件

### 3.1 组件 A · 任务级 PRD 文件

**触发条件**:
- 必须立 PRD:多文件改 / 跨小时任务 / 涉及多个项目 / 凌晨自主任务 / 任何 Doctor 不全程在线监督的任务
- 可省 PRD:单一问答 / 单文件小改(< 10 处)/ 纯研究讨论

**路径规范**:
```
~/Documents/Claude/brain/{项目名}/PRDs/{YYYY-MM-DD}_{任务简称}.md
```
或对一次性 Cowork 任务:
```
~/Documents/Claude/brain/logs/checkpoints/{YYYY-MM-DD}_{任务简称}_PRD.md
```

**核心结构**:
- frontmatter(任务名 / 立 PRD 时间 / 状态 / Doctor)
- §一 任务目标(1-2 段)
- §二 **交付标准**(checkbox 列表 · 每条**可验证**)
- §三 非交付项(写明"不在本任务范围"避免范围漂移)
- §四 状态:
  - □ 进行中
  - □ 待 Doctor 验收(全部 CC 评"我认为已达成")
  - □ 已交付(Doctor 全打 ✓)
  - □ 已取消(原因 + 时间)

### 3.2 组件 B · 三态评估纪律

**关键规则**:
- CC 对每条交付标准只能填三态之一,**永远不允许打 ✓**:
  - `[?] 我认为已达成 · 证据:{文件路径/Grep 结果/脚本输出}`
  - `[!] 未达成 · 原因:{为什么}`
  - `[~] 不确定 · 需 Doctor 判断:{具体不确定点}`
- 只有 Doctor 可以把 `[?]` 转为 `[✓]`
- 必带"证据"(可验证)— 没证据 = 等于没填

**违规检查**:CC 看到自己即将在 PRD 上打 `[✓]` → **立即停 + 改 `[?]`**

### 3.3 组件 C · 关闭路径(★ 双向闭环)

**两种合法关闭**:
1. **全 ✓ 关闭**:所有交付标准 Doctor 都打 ✓ → 状态变"已交付" · 触发 /save
2. **Doctor 取消**:Doctor 明示"取消" / "算了" → 状态变"已取消" + 原因 + 时间 · 仍触发 /save(记录未完成项)

**禁止路径**:
- ✗ CC 自动关闭(commit 即关 / 标完最后一个 Task 即关 / "我觉得交付完了"即关)
- ✗ 默认超时关闭(任务僵在那也不能默认关 · 等 Doctor)

## §二 · 交付标准(Acceptance Criteria)

> CC 三态:`[?]` 我认为已达成 / `[!]` 未达成 / `[~]` 不确定 — **CC 不允许打 `[✓]`**
> 只有 Doctor 可以把 `[?]` 转为 `[✓]`

### 文件层面

- [ ] 文件 X 已建于 ~/Documents/Claude/.../X.md
  - CC 填:`[?] 已建 · 证据:Read 命令返回 200 行内容 + ls -la 显示文件 size 12 KB`
- [ ] 文件 Y 已删除
  - CC 填:`[?] 已删 · 证据:Glob 模式不再匹配`

### 一致性层面

- [ ] Grep "旧名" 在 path/ 无残留
  - CC 填:`[?] 无残留 · 证据:Grep 结果 0 matches`
- [ ] 所有引用旧路径的文件都已更新
  - CC 填:`[?] 已更新 · 证据:Grep "old/path" 在 dir/ 找到 51 个文件 + sed 批量替换后再 Grep 0 matches`

### 功能层面

- [ ] 脚本 Z.py 能跑通输出 W
  - CC 填:`[?] 能跑 · 证据:bash 运行输出 = "..."`
- [ ] 链接 computer://path 在 Cowork 卡片能打开
  - CC 填:`[~] 我无法验证 · Doctor 需测试`

### 自审层面

- [ ] G-X1 第 5 条 self-audit 段已写(若适用)
  - CC 填:`[?] 已写 · 证据:位于 session-log §X`
- [ ] G-X1 第 7 条 checkpoint 落盘(若长任务)
  - CC 填:`[?] 已写 · 证据:progress 文件路径`

### 沟通层面

- [ ] /save 已触发(若需)
- [ ] 关键决策 / 浮现洞见已落档到 brain
  - CC 填:`[?] 已落 · 证据:位于 brain/{path}/§X`

## §六 · 与现有机制的精确联动

| 时机 | TaskList | session-log | G-X1 self-audit | G-X1 checkpoint | **PRD 交付标准(新)** |
|---|---|---|---|---|---|
| 任务开始前 | — | — | — | — | **★ 立 PRD · 写交付标准** |
| 任务执行中 | 用 | — | — | 用(progress 文件) | 不变(PRD 不变动) |
| 任务结束时 | 全 completed | 写 /save | 写 self-audit | 写最终 progress | **★ 对每条交付标准填 [?]** |
| 验收 | (无) | (无) | (无) | (无) | **★ Doctor 打 ✓ / 退回 / 取消** |
| 关闭 | (commit 即关) | (/save 即关) | (无) | (无) | **★ 全 ✓ 或 Doctor 取消才关** |

**关键观察**:
- **PRD 与 TaskList 并行不冲突**:PRD = WHAT done means;TaskList = HOW to do
- **PRD 与 G-X1 第 5 条 self-audit 是"二审"关系**:self-audit 是 CC 自审;PRD 验收是 Doctor 复审
- **PRD 与 checkpoint 是"果与过程"关系**:checkpoint 保过程不丢;PRD 保结果不漂

## §七 · 分级使用规则

| 任务类型 | PRD | 例子 |
|---|---|---|
| **必须 PRD** | ✓ | 多文件改 / 跨小时 / 跨项目 / 凌晨自主任务 / 任何 Doctor 不全程监督 |
| **建议 PRD** | 可选 | 单文件大改(>= 50 行)· 单一 skill 升级 · 跨多个工具使用 |
| **可省 PRD** | ✗ | 问答 / 解释 / 单文件小改(< 10 处)/ 研究讨论 |

**模糊判断指南**:
- 不确定 → 问 Doctor"这个任务要立 PRD 吗"
- 任务规模超预期 → 中途补立 PRD(允许)
- 任务缩到很小 → 中途撤 PRD(只能 Doctor 决定)

## §九 · 边界 / 风险 / 反例

### 9.1 不要走极端官僚化

- 问答型任务不立 PRD(开"PRD: 问 Doctor X · 交付标准:1. 回答了"是傻的)
- PRD 应该 1 页内 · 不要 3 页(超过 3 页说明任务该拆)
- 不要预先列 100 条交付标准 · 列**关键 + 可验证 + 容易漂移**的那几条

### 9.2 反例 = 不该走 PRD 的任务

- Doctor 问"朱熹现在什么重要性?" → 不立 PRD · 直接答
- Doctor 说"补 1 条 gotchas" → 不立 PRD · 直接补
- Doctor 让 CC 总结某文档 → 不立 PRD · 直接总结

### 9.3 风险

- **风险 1 · PRD 形式化**:CC 立 PRD 时把"做完任务"写成"做完任务" — 无意义。**对策**:模板里强制"可验证"字段,没证据=没填
- **风险 2 · Doctor 嫌烦不审**:Doctor 累不审 PRD → 任务僵在"待验收"。**对策**:Doctor 可以一句话"全 ✓"批量验收 + 阶段 3 加 brain-save Step 5.5 自动提示"待 Doctor 验收"
- **风险 3 · CC 把 [?] 当 [✓] 用**:CC 心理上"我填了 [?] 就完了" → 实际上等 Doctor 审。**对策**:模板里加强提示 + brain-save 回报模板必须包含"待 Doctor 验收"字样
- **风险 4 · 历史任务追溯麻烦**:已经做过的任务不必反推 PRD。**对策**:只新任务用

## §十 · 待 Doctor 裁定的子决策(★ 这是核心)

### 决策 1 · 立 PRD 的"触发权"归谁

- **A · Doctor 主动说**:"这个任务立个 PRD" → CC 立(默认安全)
- **B · CC 主动提**:CC 判断任务规模超阈值 → CC 提"建议立 PRD" → Doctor 同意
- **C · 默认必立 PRD 除非 Doctor 说省**:大胆推进 · 但可能过度

(我推荐 **B** — 给 CC 启动权,Doctor 一票否决权)

### 决策 2 · PRD 文件位置

- **A · 项目内部**:`~/Documents/Claude/brain/{项目名}/PRDs/`(分散 · 与项目耦合)
- **B · brain 中央**:`~/Documents/Claude/brain/PRDs/`(集中 · 易追踪)
- **C · checkpoints 目录**:`~/Documents/Claude/brain/logs/checkpoints/`(与 G-X1 checkpoint 同位)

(我推荐 **C** — 与 checkpoint 同栖 · 大任务的 PRD + progress 同目录方便)

### 决策 3 · CC "✓" 权限的硬约束程度

- **A · 软约束**:只在文档约定 · 靠 CC 自觉(简单 · 但本次 G-X2 已证明"靠自觉"会失效)
- **B · 模板硬约束**:模板里 checkbox 默认是 `[ ]` · CC 只能改成 `[?]/[!]/[~]` 三态 · 看到 `[✓]` 就报错回退
- **C · skill 级硬约束**:开发一个 lint 脚本 · 跑前检查 PRD 里有没有 CC 误打 `[✓]`
