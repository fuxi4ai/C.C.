# 视频方法论 × Doctor 开发流程 — 对照与优化设计提案

> **一句话结论**：视频的"毒舌产品经理 5.0"与你的 brain 体系**独立收敛到了同一套核心**（skill 分阶段、文档驱动跨阶段上下文、独立审查子 agent、信号→规则自进化、目标+验收的循环）。在**治理维度你已经领先视频**（PRD 三态验收、数据真实性铁律、propose-then-confirm）。所以这不是"照着视频重做流程"，而是**挑视频的 3 个净增量补进你的体系，并明确拒绝 2 个与你价值观冲突的点**。

---

## §1 · 视频方法论拆解（毒舌产品经理 5.0）

作者的软件开发工作流，6 条主线 + 3 条经验：

1. **轻流程**：skill 不写分步操作手册，只写「何时用 / 什么算合格 / 什么红线」；能用程序判断的事（如"代码没过审不许收工"）做成**自动门禁**，不写进散文规则反复念。
2. **6 阶段流水线，每阶段一个 skill**：需求收集（prd-spec-builder，"毒舌"=禁阿谀奉承、把模糊想法逼成需求文档）→ 设计规范（design-brief-builder）→ 设计图（design-maker，可选）→ 开发计划（dive-planner，拆成可独立验收的 face）→ 开发（dive-builder，闭环：每个任务派独立子 agent 审查，过了才算完）→ 发布（release-builder，隐私审计+打包）。加修 bug、造技能等共 11 个 skill。
3. **文档驱动跨阶段上下文**：每阶段产出一份文档，下一阶段拿着它干；跨阶段不丢上下文靠的就是这些文档。
4. **Agent 编排**：默认**单主 agent 从头干到尾**；只有两种情况派子 agent——(a) 需要一颗干净的新脑子（如审查员），(b) 几块互不依赖、可并行的活。子 agent 是"另一颗脑子不是另一个文件夹"，上下文整份带走、结果回主 agent 合并拍板。
5. **自进化**：你纠正/批评它，它存成一条条 signal；下次新开 session，派一个子 agent（evolutionRunner）消化这些 signal，抽象成规则修改建议（替换/补充/新增），逐条问你要不要改；**双向**——没用过的规则它也主动建议删。
6. **go 机制**：一条 go = 目标 + 完成标准 + 验收方式；AI 拿到就进循环直到达成。写了 go-creator skill 让 **AI 替你起草 go**（因为它在场最懂状态）。配套纪律：**一个 session 只开发一个功能**，上下文小而干净，模型注意力最佳。
7. **三条经验**：① 别把能力拆成一堆小工具，给完整通用能力让它自己想办法（"聪明是放出来的不是设计出来的"）；② 规则别提前写，一条规则必对应一次真实失败，不用了就删；③ 把劲花在**设计循环**上，不在写提示词/盯梢上——人定目标、AI 跑循环。

## §2 · 现状对照：你已经有的 vs 视频的

| 视频组件 | 你已有的对应物 | 成熟度判断 |
|---|---|---|
| go = 目标+完成标准+验收，进循环 | **PRD 交付标准机制 G-X4**（三态评估 `[?]/[!]/[~]`、可机器验证、Doctor 打 ✓）+ **长任务 checkpoint 循环**（CC长任务防错指南） | ✅ 你更严：多了"可验证=客观"硬约束 + Doctor 验收闸 |

| 自进化 signal→rule（子 agent 消化） | **GOTCHAS 自动记错 + 经验库半自动记成功 + brain-consolidate 蒸馏** | 🟡 同源，但缺"signal 队列 + 开 session 自动消化"的主动形态 |
| 独立审查子 agent（没参与开发） | **句芒 `project-review`**（PRD对齐+代码质量+精神洞察三层，审时不参与产出、只读不写） | ✅ 几乎一模一样，你已落地 |
| skill 分阶段流水线 | 部分：conch（项目维护）、skill-creator；**无"需求→设计→计划→开发→发布"成套** | 🟡 部分缺口（但你主体非 product-dev，见 §3） |
| 文档驱动跨阶段上下文 | **brain vault L0/L1/L2 分级 + architecture/决策记录/features** | ✅ 你更系统 |
| 单主 agent + 按需子 agent | **数灵 subagent 编排 + 数灵转移设计方案**（运行层+记忆层双挂载） | ✅ 你更细致 |
| 轻流程（skill 只写边界） | 你的 skill 偏厚（brain-save 步骤细）、GOTCHAS 很厚 | 🟡 张力点，可选优化 |
| 能判定→自动门禁 | check-cascade.py、validate-frontmatter.py、checkpoint 脚本 | ✅ 部分已有 |
| **默认全自动 + 全放开权限** | **刻意相反：propose-then-confirm + Doctor 打勾 + 数据铁律** | ⛔ 价值观冲突，见 §3 |

**核心观察**：你独立建成了视频鼓吹的约 80%，且在"防漂移/防假数/人在环"这条治理轴上**比视频成熟**。视频的真正净增量很窄——主要是 go-creator 的"省力"和 signal 自进化的"主动"两点。

## §3 · 优劣判断

### 可借鉴（建议采纳）

1. **go-creator 思路 → 增强 G-X4**。让 CC 在大任务起手时，结合当前上下文**自动起草整份 PRD**（目标/完成标准/验收方式），你只审一眼、改一改、点同意。这是 G-X4 §前瞻"打勾权下放"路线的自然前置——不下放打勾权，但下放"起草权"。低风险、契合度高。

2. **skill 三段式骨架 + "能判定的做硬门禁"**。把 skill 写法规范成「何时用 / 什么算合格 / 什么红线」，把"能用程序判断的"做成自动门禁而非散文规则。这条直接治你**GOTCHAS/规则越写越厚**的隐忧，与你 G-X4"可验证=客观"、check-cascade 自动门禁同源。

3. **"一个 session 一个功能、上下文小而干净"纪律显式化**。你已有 L0/L1/L2 和 brain-resume，但"一功能一 session、靠文档接力"这条纪律没写死。显式化它，正好缓解你 G-X1/长任务防错反复担心的**上下文耗尽**。

4. **signal 队列 + 开 session 主动消化**。比你"经验库半自动、/save 时才提议"更主动：纠正即记一条 signal 进 inbox，新 session 由子 agent 消化成规则提案。形态上接近你已设想但未完全工程化的东西，且天然嵌进 brain-resume。

5. **三条经验里的两条你其实已立，可作显式背书**：规则从失败里长（=你 GOTCHAS 一坑一条）、删不再用的规则（=brain-consolidate 修剪）。第三条"把劲花在设计循环不在监工"——可作为对 G-X4 演进路径的提醒：在**客观可验证 + 可回滚**的工程任务上，逐步试点打勾权下放。

### 不合适（建议明确拒绝）

1. **⛔ 默认全自动执行 + 全放开权限**。直接违反 propose-then-confirm 与数据铁律。视频语境是"我自己用、低风险、可回滚的小工具"；你的项目大量是**研究/分析/投资决策**，假数与漂移代价极高（白泽占位案、哲学史★级膨胀案都是活体反例）。→ 维持人在环；仅在"客观可验证+可回滚"的纯工程任务上，可考虑开局部白名单（§5 决策点）。

2. **⛔ demo-first 脚手架占位跑通**。视频 webcut 用样本/测试 skill 跑通——你有活体血教训（白泽 `run_full_analysis` 硬编码占位价险进报告）。明确不引入。

3. **🟡 "轻流程"一刀切要防误读**。视频说"写越细越压模型上限"，但你的很多厚规则恰恰是**环境/安全硬约束**（数据铁律、git 沙箱禁忌 G-X2、FUSE 盘 sqlite 坑、Cowork 落点规则），不是"压上限的步骤说明"。判别尺：压「怎么做(how)」可以轻，但「什么算好 / 什么红线 / 环境硬坑」必须厚。视频作者自己也澄清了"不是写薄，是删 how、留 what+红线"——采纳这个精确版，拒绝口号版。

4. **🟡 全套 11 个 product-dev skill 照搬**。你的主体是研究知识工作，不是连续造 app。design-brief-builder / design-maker / release-builder 这套 UI 产品线大部分用不上。只挑可迁移的**元能力**（go-creator、自进化、审查闭环），不搬整条产品流水线。

5. **⛔ cloud design / cursor 等具体工具链**。与你 Cowork + APIYI + brain 环境不符，跳过。

### 张力核心（一句话）

视频 = **单人快速造可回滚的小工具，最大化自主性**；你 = **多项目长期研究资产，最大化可信度与不漂移**。优化目标不同：自主性维度谨慎借，工程纪律维度你本就领先。

## §4 · 是否需要新 skills（设计判断）

**不需要**新建一整套 product-dev skills。建议三个候选改造，每个都给出"落 skill 还是落规则/模板"的取舍，由你裁定：

- **(A) skill 写作规范**：编码"三段式骨架 + 能判定做硬门禁 + 删 how 留 what"。→ 倾向落 **references/ 文档**（也可注入 skill-creator）。低风险。
- **(B) go-creator 能力**：CC 大任务起手自动起草 PRD。→ 落 **brain-save 的一个前置步骤**，或独立新 skill `brain-prd`。这是 G-X4 阶段 3"集成到 skill"的兑现。
- **(C) signal→rule 自进化**：纠正即记 signal、新 session 消化成规则提案。→ 倾向**扩展 brain-resume + brain-consolidate**（resume 时消化 signal inbox），而非新 skill；若要独立可叫 `brain-evolve`。
