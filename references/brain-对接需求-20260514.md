# Brain Vault 对接需求 v1.0

> **🗄️ 归档说明**：本对接需求 21 条已全部完成（见 brain/.tools/、brain/.skills/、brain-vault-dashboard artifact）。本文档保留作历史，不再是 active spec。

> 给 Doctor 的实施清单。读完即可分头行动：**自建**部分自己写代码 / 配置；**找现成**部分按关键词搜 skill 或插件。

## 0. TL;DR

Brain Vault 当前是**纯脚手架**（目录 + 模板 + 规则文档），还没有任何实际运转的工作流。本文档把它从"骨架"推到"可用产品"所需要的全部对接点列清楚，分 7 个模块、共 21 条需求，每条标注：

- **类型**：🔨 自建 / 🔍 找现成 / 🧩 混合
- **优先级**：P0（不做则 brain 不能跑） / P1（关键体验） / P2（锦上添花）
- **验收标准**：完成的客观判据

1. **`/resume` `/save` `/note` 是文字规则，不是真实可执行命令**
2. **`logs/` 完全是空的**，没有过往 session 可恢复

7. **没有孤儿笔记（orphan note）检测**
8. **没有全文搜索**——找一条笔记只能 `grep`
9. **没有语义检索 / memory_store 集成**——文件方案目前无嵌入
10. **`Claude/Projects/` 里 7 个项目和 brain 关联只有"渊图"**，其余无登记
11. **锚点触发（dva / 龙鱼五力 / 自检 / 天工开物）没有强制执行机制**
12. **没有 GOTCHAS.md**，旧方法论的错误沉淀机制未承接
13. **brain 自身不在版本控制下**（无 git history，丢就丢）
14. **没有跨设备同步策略**（如果有第二台设备）
15. **CC 进入工作目录时不会自动触发 START_HERE / CLAUDE.md 读取**，靠 Doctor 主动提示

## 2. 详细需求

### 模块 A · 命令系统（让 `/resume` `/save` `/note` 真正可执行）

**问题**：CLAUDE.md 写了 3 条 slash 命令，但它们是给 CC 的散文描述，没有 skill 文件承接。每次会话都要 Doctor 提醒 CC 去读规则。Cowork 体系里，命令真正生效必须是**安装好的 skill**，或者**钩在 CLAUDE.md 顶部的强约束**。

#### REQ-A1 · `/resume` 实现 · P0 · 🔨 自建

- **行为契约**：CC 在用户输入 `/resume`（或听到"恢复上下文"等同义指令）时，必须：
  1. 列出 `logs/` 下按时间倒序的前 3 个文件
  2. 全部 Read 后输出摘要：上次工作的项目、关键决策、遗留待办
  3. 若工作项目可识别，再 Read `{project}/architecture/decisions.md` 与 `progress.md`
  4. 末尾给出"建议下一步"的 3 条
- **形态选择（任选其一）**：
  - **a)** 在 `Claude/brain/.claude/commands/resume.md` 写 slash 命令（如 Claude Code 支持）；或
  - **b)** 写成 Cowork 插件 skill，安装后 `/resume` 触发；或
  - **c)** 在 `CLAUDE.md` 顶部加一段"CC 收到 `/resume` 字面字符串时强制执行以下流程"，靠 prompt 约束。
- **验收**：在干净 session 里说 `/resume`，CC 自动读最近 3 个 log 并输出摘要，**不需要 Doctor 进一步指示**。
- **依赖**：先得有日志（见 REQ-A2），否则没东西恢复。

#### REQ-A2 · `/save` 实现 · P0 · 🔨 自建

- **行为契约**：用户输入 `/save` 时 CC 必须：
  1. 用 `templates/session-log.md` 模板填充本次会话
  2. 写入 `logs/YYYY-MM-DD-{主题}.md`（主题由 CC 总结）
  3. 自动填充：完成的工作、做出的决策、遗留问题、相关 wikilinks
  4. 若指明项目，同步更新 `{project}/progress.md`
  5. 回报：日志路径 + 一句话摘要
- **关键设计点**：
  - 主题如何抽取（CC 自动总结 vs 用户指定）
  - 同一天多次 `/save` 是新建文件还是追加（建议新建，文件名带 hh-mm）
  - 模板变量替换（`{{date}}` `{{project}}` `{{duration}}`）由谁完成
- **验收**：会话末尾说 `/save 渊图`，`logs/2026-05-14-渊图.md` 出现，包含本次有意义的记录。

#### REQ-A3 · `/note [主题]` 实现 · P1 · 🔨 自建

- **行为契约**：用户 `/note 知识图谱-节点类型` 时 CC 必须：
  1. 用 `templates/default-note.md`（或 `resource.md`，由 CC 判断）填充
  2. 写入 `inbox/{主题-kebab-case}.md`
  3. CC 进入"采集模式"：用户口述/补充，CC 实时整理结构
  4. 内容成熟时建议提升到 `permanent/`
- **验收**：`/note 测试主题` 后 `inbox/测试主题.md` 出现，含正确 frontmatter，CC 进入对话采集态。

### 模块 B · 笔记治理（Zettelkasten 纪律自动化）

**问题**：CLAUDE.md 定下了纪律（必须 frontmatter、必须 wikilink、不能 markdown 链接、permanent 必须 ≥2 个 wikilinks），但**靠人遵守=不遵守**。

#### REQ-B1 · Frontmatter 验证脚本 · P1 · 🔨 自建

- **要求**：扫描 brain/，对所有 .md 文件检查：
  - 是否有 YAML frontmatter
  - 必填字段（title / tags / created / updated / status / type）齐全
  - `status` 取值合法（active / archived / draft）
  - `type` 取值合法（permanent / fleeting / log / resource / decision / spec）
- **形态**：Python 或 Node 脚本，放 `brain/.tools/validate-frontmatter.py`，每次 `/save` 前调用，或独立 `validate` 命令。
- **验收**：故意写一条缺 frontmatter 的笔记，脚本能精确报错并指出文件 + 行号。

#### REQ-B2 · Wikilink 解析与回链（backlink）索引 · P1 · 🔨 自建

- **要求**：
  - 扫全 vault，提取所有 `[[xxx]]`
  - 解析每个 wikilink 指向的文件（按文件名匹配，不分目录）
  - 生成 `brain/.index/backlinks.json`：`{被链接的笔记: [链入它的笔记列表]}`
  - 当 CC 进入某条笔记，能反查"谁链到我"
- **形态**：脚本生成 JSON 索引，CC 通过 Read JSON 查询（轻量方案）；或塞进 Obsidian 兼容索引（重方案）。
- **验收**：建两条互链笔记，运行索引脚本后 backlinks.json 正确反映双向关系。

#### REQ-B3 · 孤儿笔记检测 · P2 · 🔨 自建

- **要求**：所有 `permanent/` 笔记必须有 ≥2 个 wikilinks（出 + 入合计），低于阈值标记为孤儿。
- **形态**：`brain/.tools/find-orphans.py`，输出列表。
- **验收**：能列出孤儿笔记并给出提升建议。

#### REQ-B4 · 全文 / 标签搜索 · P1 · 🔍 找现成 或 🧩 混合

- **要求**：按关键词、标签、frontmatter 字段搜索 brain。
- **可选方案**：
  - **轻**：用 `ripgrep` 即可，写个 shell alias
  - **中**：装一个 markdown 检索 skill（GitHub 关键词：`obsidian-search` `markdown-vault-search` `zettelkasten-cli`）
  - **重**：上 Obsidian 本地客户端打开 brain/，所有功能现成（搜索、回链、图谱、tag）
- **建议**：直接装 **Obsidian** 打开 `brain/` 作为 vault，B2/B3/B4 一并解决。CC 这边只需要文件读写，Obsidian 负责人机交互层。
- **验收**：能在 5 秒内定位任何已存在的笔记。

### 模块 C · 数据通道（让 Chats / Graphify / Projects 真正灌进来）

#### REQ-C1 · Claude Code 对话导入 · P1 · 🔍 找现成

- **目标**：把 Claude Code 历史 session 自动归档进 `chats/code/`。
- **现状**：目录在那，但没有任何文件，没有导入工具。
- **关键词搜索**：`claude-code-export` `claude-code-session-archive` `claude-code-history-importer`
- **参考**：原始 GitHub `lucasrosati/claude-code-memory-setup`（你的 brain 系统来源）应该有相关脚本
- **验收**：跑一次导入，`chats/code/` 出现按 session 切分的 markdown，每条有正确 frontmatter（含 chat ID、日期、模型、tokens）。

#### REQ-C2 · Claude Web/App 对话导入 · P2 · 🔍 找现成

- **目标**：从 Claude.ai 网页/桌面客户端导出对话进 `chats/web/`。
- **关键词搜索**：`claude-web-export` `claude-conversation-exporter` `claude-ai-archive`
- **注意**：Claude.ai 自身有导出功能（Settings → Privacy → Export data），但导出是 JSON，需要转 markdown 切分。
- **验收**：导出 → 转换 → 落到 `chats/web/{date}-{title}.md`，CC 可直接读。

#### REQ-C3 · Graphify 集成 · P2 · 🔍 找现成

- **目标**：对代码项目跑 Graphify 生成 `graph.json`，CC 优先查图理解代码结构。
- **当前**：`graphify/渊图/` 子目录存在但是空。
- **关键词搜索**：`graphify` `code-knowledge-graph` `repo-graph-generator` `tree-sitter-graph`
- **CLAUDE.md 已定的查询顺序**：graphify → vault → 源代码
- **验收**：对某个代码项目跑一次，`graphify/{project}/graph.json` 生成，CC 能基于它回答"X 函数调用了哪些模块"之类问题。

#### REQ-C4 · `Claude/Projects/` ↔ `brain/` 关联 · P0 · 🔨 自建

- **问题**：Projects 下有 7 个项目（DVA / O MY HTML / Optical communication / 司南 / 政治经济学 / 海螺姑娘 / 龙鱼五力），brain 里只登记了"渊图"。结构上的鸿沟：
  - 渊图直接是 `brain/渊图/` 的子目录
  - 其他 7 个项目实体在 `Projects/{name}/`
  - 两套路径不统一，CC 找项目状态时不知道去哪
- **决策需要**：
  - **方案 A**：所有项目都在 `brain/{name}/` 下放一份"项目档案"（architecture / decisions / progress / logs），代码和资产仍在 `Projects/{name}/`。两个路径靠 frontmatter `code_path` 字段关联。
  - **方案 B**：把 brain/{name}/ 整个并入 Projects/{name}/.brain/，每个项目自带一份 brain。
  - **方案 C**：保持现状，brain 只管"会议室级别"的项目（渊图）；其他项目自管。
- **建议**：方案 A——统一进 brain，便于 CC 一站式 `/resume`；代码资产在 Projects 不动。
- **配套工作**：写 `register-project.sh` 脚本，传项目名自动生成 brain 骨架 + 项目档案模板。
- **验收**：跑 `register-project.sh 龙鱼五力`，`brain/龙鱼五力/{architecture,pipeline,data,features,logs}/` + 一份 README 自动生成，CLAUDE.md 项目列表自动追加。

### 模块 D · 记忆引擎（LanceDB 决策）

**背景**：旧方法论（`4AI/文件系统管理经验.md`）强烈推荐 **memory-lancedb** 插件，做语义检索而非纯关键词。但 brain 当前的设计是纯文件 + Zettelkasten。

#### REQ-D1 · `memory_store` / `memory_recall` 决策 · P1 · 🧩 混合

- **决策选项**：
  - **a)** 双轨：文件为主存储，LanceDB 作为副本索引；每次 `/save` 写文件 + 写向量
  - **b)** 单轨：只文件，靠 ripgrep / Obsidian 解决检索
  - **c)** 反过来：LanceDB 为主，文件只是导出
- **建议 a**：文件是真理之源（人类可读、git diff 友好），LanceDB 是加速索引（可丢可重建）。
- **关键词搜索**：`memory-lancedb` `claude-memory-vector` `mcp-memory-lance`
- **验收**：决策定下来后，写进 `CLAUDE.md` "Memory策略"段；如果走 a，建立第一次同步管道。

#### REQ-D2 · 语义检索 API · P2 · 🔍 找现成

- **目标**：CC 问 "找跟「期权波动率」相关的笔记"，能返回 top-k 而不只是关键词命中。
- **依赖**：REQ-D1 决策后才能开工。
- **验收**：典型语义查询能跨笔记找到相关条目，即使关键词不完全一致。

### 模块 F · 规则守卫（承接旧方法论）

#### REQ-F1 · 锚点触发机制 · P0 · 🔨 自建

- **规则**：当对话中出现以下关键词，CC **立即停下当前任务**，先完整读取该项目/技能全部文件，再继续响应。
- **现有锚点**：

  | 关键词 | 触发动作 |
  |---|---|
  | `dva` | 加载 DVA 项目全套（`Projects/DVA/` + `brain/DVA/`） |
  | `龙鱼五力` | 加载龙鱼五力分析框架全套 |
  | `自检` | 触发严格用户视角标准化检查流程 |
  | `天工开物` | 加载 `Vault/天工开物.md` 设计项目启动咒语 |

- **形态选择**：
  - **a)** 写进 `CLAUDE.md` 顶部强制段，CC 进会话先读
  - **b)** Cowork hook：用 user-message-prefix 拦截关键词触发自动 Read
  - **c)** 写成 `锚点表.md` 单独文件，CLAUDE.md 引用
- **建议 c**：单独维护，CLAUDE.md 只引一句"见 anchors.md，规则强制"。
- **验收**：在干净 session 输入 "处理一下龙鱼五力数据"，CC 自动读完龙鱼五力相关文件再回应，无需 Doctor 提示。

#### REQ-F2 · GOTCHAS 机制 · P1 · 🔨 自建

- **要求**：每个项目（及未来每个自建 skill）都有一份 `GOTCHAS.md`，记录踩过的坑。
- **统一格式**（沿用旧方法论）：

  ```markdown
  ## [ERR-YYYYMMDD-XXX] 简要描述
  **状态**: ⏳ 待解决 | ✅ 已解决
  **优先级**: 🔴高 | 🟡中 | 🟢低
  **触发时间**: ISO 8601
  **触发场景**: ...
  **错误信息**: ...
  **解决方案**: ...
  **预防措施**: ...
  ```

- **配套**：项目骨架生成器（REQ-E1）默认创建空 GOTCHAS.md。
- **验收**：所有 brain 注册项目目录下 GOTCHAS.md 都存在。

### 模块 G · 备份、版本控制、同步

#### REQ-G1 · Git 版本管理 · P0 · 🔨 自建

- **要求**：`brain/` 整个目录纳入 git。
- **关键决策**：
  - 仓库是 brain/ 本身（单独 repo）还是 `~/Documents/` 顶层（统一 repo）？建议 brain/ 单独

- **方案**：iCloud Drive / Dropbox 自动同步 + git remote 双保险。
- **验收**：删除本地 brain/ 后能从远端恢复完整。
