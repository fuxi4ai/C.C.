## §二 · 交付标准(Acceptance Criteria)

> ⚠️ **CC 填写规则**:
> - `[ ]` = 默认未开始 · CC 不能直接改 `[✓]`
> - `[?]` = CC 认为已达成 · **必带证据**(具体路径 / Grep 结果 / bash 输出 / 文件 size 等)
> - `[!]` = CC 未达成 · **必带原因**
> - `[~]` = CC 不确定 · **必写需 Doctor 判断的具体点**

- 任何情况下 CC 看到自己将打 `[✓]` → 立即停 + 改 `[?]`

**未来(实战 3-5 次后 · 阶段 3 评估)**:

- 防漂移依赖**PRD 标准本身的客观性筑基** · 不是依赖 CC"自觉"
- 任何在阶段 1-2 偷跑打 ✓ 的行为 = 重大违规 · 落 GOTCHAS

## §七 · 模板使用规范(给 CC)

### 立 PRD 时
1. CC 主动判断任务规模 → 提"建议立 PRD" → Doctor 同意
2. 复制本模板到 `~/Documents/Claude/brain/logs/checkpoints/{date}_{name}_PRD.md`
3. 填 §一(目标 + Doctor 原话 + 规模估算)+ §二(交付标准 · 必须可验证)+ §三(非交付项)
4. CC **不能填 §二 的 `[ ]` 之外的状态** · 等开始执行后才能改

### 任务执行中
- 每达成一条交付标准 → 立即改 `[ ]` → `[?]` + 填证据(不要等任务结束一次填)
- 中途发现某条达不到 → 改 `[!]` + 原因 + 提 Doctor
- 中途发现某条不确定 → 改 `[~]` + 列出 Doctor 需判断的具体点
- 任务范围扩大 → 不要私自加交付标准 → 提 Doctor 审批 + 改 §一

### 任务结束时
1. 所有交付标准都填了三态之一 → 把 §四 状态从"进行中"改为"待 Doctor 验收"+ 时间
2. 触发 /save → 在 brain-save 回报里贴 PRD 文件路径 + 状态
3. 等 Doctor 验收
4. **不能在 §四 自打"已交付"**

### 任务取消时(Doctor 操作)
- Doctor 写"取消"+ 原因 → §四 状态"已取消"+ 时间 + 原因
- CC 触发 /save 记录未完成项
