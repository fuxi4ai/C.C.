## 完成的工作

- `/resume` 恢复上下文：认领全局偏好镜像（敬语「您」等），读近 3 篇 log（司南朝向/PEC/白泽弹性v2），输出结构化摘要。
- **主库「太大/沙箱出问题」诊断**：厘清渊图有两个「主文件」——`index.json`（248KB 报告索引，非本体）vs `mapping/行业知识图谱_完整数据库.json`（8.4MB / 22.3万行 / 3122节点+3676边，真·KG 主库）。主库沙箱 `json.load` 0.04s 秒解、结构完整，**未损坏**；问题不在文件坏/沙箱坏，在**取法**——22万行远超 Read 2000 行窗口、`cat` 全量撑爆上下文。22万行是 `indent=2` pretty-print 导致的体积虚高。
- **「更新是否需完整读取」诊断**：抽取步（kg_ingest）`build_node_reference` 已封顶 `MAX_REF_NODES=120`（关键词打分 top-120），模型每次最多看 120/3122 节点、只吐 patch，**从不整读**；合并步（kg_merge）`load_json` 全图入内存（0.04s，碰不到上下文）按 ID 索引就地改。结论：自动更新路径下「太大」是非问题；真正要盯的是 120 封顶的**召回风险**（漏配→重复节点，_canon 碰撞集只拦 alias 张冠李戴、不强塞漏掉节点，故留 dedup_check.py）。
- **无损优化（Doctor 批「只归档历史簇」方向）**：勘出 `mapping/` 612MB 里只有 8.4MB 是 live；其余 ~600MB + root `backups/` 366MB 全是历史冗余、且**全被 .gitignore 忽略**（`mapping/*.bak_*`、`mapping/行业知识图谱_v*_*`、`_v*_*`、`archived/`、`backups/` 等），0 个 git-tracked。
- **三簇 tar.gz 归档（Doctor 批「压缩后移除松散拷」）→ `archived/2026-07-10/`**：
  - `mapping_bak.tar.gz` 64 项 → 56M
  - `mapping_dated_snapshots.tar.gz` 65 项 → 30M
  - `root_backups.tar.gz` 73 项 → 57M

## 做出的决策

| 决策 | 原因 | 影响 |
|------|------|------|
| 无损优化＝归档历史簇，**不**重排 live 主库 | 整读进上下文即便压缩仍过大（该查询式取）；唯一不毁 git diff 的 JSONL 是重构非无损；`indent=2` 正是 kg_merge_safe「出 diff 给您审」安全叙事的可读性来源 | 主库一字不动；退而以 jq/python 查询式取数 |
| tar.gz 压缩后移除松散拷（非纯 mv） | 纯 mv 同文件系统不回收磁盘；tar 内保全＝无损，解压即还原 | 净回收 ~780MB，143MB tar 留档 |
| 归档去处 `archived/2026-07-10/` | 项目现成惯例（.conch_audit.log 一直归此），且 archived/ 已 gitignore | 不撑大 git 仓 |
| 沙箱删既有文件走 `allow_cowork_file_delete` 授权 | 沙箱对删用户文件有权限闸，报 EPERM | 授权后由 CC 精确删；亦可退回贴命令给 Doctor 终端 |
