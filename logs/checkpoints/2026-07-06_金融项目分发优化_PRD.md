# PRD · 金融项目分发优化（首期：烛照九阴日报分发版）

## §一 · 任务目标

**动机**：四个内部可视化项目（烛照九阴·日报 / 白泽大宗·周报 / 龙鱼五力·标的表+边际变动 / 渊图·星空）准备面向读者分发。内部产物依赖数据库、含核心算法痕迹与内部代号，不能直接示人。需要一条**增量的"分发版"产线**：脱库、脱敏、自包含、面向非专业读者，且不动任何既有产物。

**范围（首期 MVP）**：以烛照九阴·日报为试点，建成"分发版"构建全流程（构建脚本 + 脱敏配置 + QA 校验 + 独立定时任务），并沉淀为可复用到其余三项目的范式文档。先电脑端跑通，再做移动端适配（两者都是硬验收）。

**Doctor 原始指令**（逐字引用）：
> "现在进行一个'金融项目分发优化'的架构调整。先记录一个PRD并多次对齐：1、目的：我们的烛照九阴·日报、白泽大宗·周报、龙鱼五力·常更标的表及分数边际变动、渊图星空可视化，这四个'可视化项目'，要准备面向读者了。所以，要进行'以分发为目的的优化'。2、要求：1）核心算法保密；2）数据库隔离（只按时分发不依赖数据库的html）；3）美化（e.g.图片、字体等）元素内置；4）原始Artifacts等不变，此优化及产物为增量项目；5）保密前提下的'易懂'——读者大部分为非专业人士；6）核心保密前提下的科学可信度——适当披露权威信源（如帕米尔研究可披露为'行业专家电话会议资料'、可披露FRED为'美联储XX数据'、可披露国外Tier1投行研报等）；7）核心保密前提下，可适当披露权重及评分标准。"
>
> （增量需求）"用户90%为手机端阅读，所以要对手机端进行适配。可以先跑通电脑端。"

**对齐结论摘要**（三轮 AskUserQuestion，2026-07-06）：
| 维度 | 裁定 |
|---|---|
| 载体 | 单文件自包含 HTML（字体子集/图片全内嵌，无外部依赖） |
| 首期范围 | 先试点一个打样 → **烛照九阴·日报** |
| 生成方式 | **独立定时任务**（不动既有日报班；接受 G-X51 无人值守面 +1，以防悬挂前置+哨兵对冲） |
| 对外署名 | **中性对外品牌名** → **观潮·市场日评**（Doctor 2026-07-06 定） |
| 买卖信号 | **包装为分数**呈现，不出现"买入/卖出"类操作字样，读者自行理解 + 显著风险提示 |
| 分数粒度 | **取整到个位**（Doctor 2026-07-06 定） |
| 任务时刻 | **工作日 08:30**（Doctor 2026-07-06 定；实建 08:32） |
| 移动端 | 90% 读者为手机端，移动端适配为硬验收；实施顺序 电脑端 → 手机端 |

**任务规模估算**：
- 预计新建文件数：8–12（实际：构建/QA 脚本 ×2、dist 配置 ×3+md5基线、范式文档、样张、定时任务 SKILL、session log）
- 预计修改既有文件数：**0**（增量铁律；烛照九阴.md frontmatter「最后活跃」按 E-3 例外更新）
- 预计耗时：3–6 小时（电脑端）+ 1–2 小时（移动端适配）
- 涉及项目：烛照九阴（试点）· 跨项目范式沉淀
