# 设计稿 v2 · 受益公司库领域分库

> 本稿只定**架构与口径**，不落库、不动文件。Doctor 审定后再出逐文件实施清单 + 二次确认才动手。
> v2 变更：由 v1「合并成一张表」改为「领域分库·各司其职」（Doctor 2026-06-29 定调）。

---

## 一、缘起与目标

**割裂现状**：受益公司散在两库——白泽大宗（涨价驱动）一拨、龙鱼库（主动点名 + 渊图浮现）一拨，看全集要查两处；且白泽存了一份龙鱼评分镜像，两边漂移。

**目标（Doctor 定调）**：不是把两库塞成一张表，而是**按领域把职责切清**：
- **龙鱼库 = 公司级真源**（详细基本面/六维评分）；
- **白泽库 = 大宗涨价数据真源**（品种行情 + 受益关系）；
- 白泽跑出的受益公司 → 龙鱼五力跑分 → 落龙鱼库；白泽要用个股数据时**调用龙鱼库**，自己只留轻量受益关系。

## 二、地基事实（实测 2026-06-29）

| 口径 | 标的数 |
|---|---|
| 白泽受益面带码（`fundamentals` ∪ `business_breakdown`） | 29 |
| 龙鱼 records | 86 |
| 交集 | 29 ｜ **只在白泽 0** ｜ 只在龙鱼 57 |

**龙鱼库已是白泽受益公司的严格超集**——白泽 29 家全部已在龙鱼库。所以"让白泽受益公司进龙鱼库"对存量 29 家**已物理完成**，本案主要是**定规则 + 砍冗余 + 接调用**，不是大搬迁。

## 三、决定的架构（职责单一）

```
┌─────────────────────────────┐        ┌──────────────────────────────┐
│  白泽库 business_breakdown.db │        │  龙鱼库 龙鱼-标的分析库/        │
│  = 大宗涨价数据真源            │        │  = 公司级真源（详细基本面）      │
│                              │        │                              │
│  · 品种行情/涨价(价格·同比)    │        │  · 六维评分/total/rating       │
│  · 受益关系(品种×公司:        │ 调用→  │  · thesis/engine_facts        │
│    纯度/弹性/relation/营收占比) │ ←跑分  │  · analyses[] 时间序列          │
│  · 轻量受益公司名单(品种→ts_code)│       │  · 持仓 tag(holdings 单点)     │
│  ✗ 弃 stock_scores 镜像       │        │  PK = ts_code · git 可diff      │
└─────────────────────────────┘        └──────────────────────────────┘
```

**数据流**：
1. 白泽引擎识别某品种涨价的受益公司 → 写入白泽**受益关系表**（品种, ts_code, 纯度, 弹性, relation, 营收占比）。
2. 新增受益公司（白泽新出的 ts_code）→ 推入龙鱼五力引擎跑六维 → `record_writer` 落龙鱼库 `records/`。
3. 白泽周报渲染「龙鱼六维」列 → 按 ts_code **实时调龙鱼库**取 `six_total/rating/thesis`，不再读自存镜像。
4. 烛照九阴 / CC / 其他消费方 → 经公共只读模块按需取：公司面查龙鱼、受益关系/品种行情查白泽。

## 四、字段归属（node vs edge·本案关键裁定）

| 字段 | 性质 | 归属 | 理由 |
|---|---|---|---|
| 六维评分 / total / rating / thesis / engine_facts(PE/PB/ROE/市值) | 公司**节点**属性 | **龙鱼库** | 一公司一套，天然挂公司节点 |
| 纯度 / 自产 / 长单 / 需求弹性 / 营收占比 / relation | 品种×公司**边**属性 | **白泽库** | 一公司可多品种、各一套；是白泽引擎产物、属"涨价受益逻辑" |
| 品种行情（价格/同比/涨幅） | 品种属性 | 白泽库 | 白泽本职 |
| node_id / industry / track | 公司节点属性 | 龙鱼库 | 公司画像 |
| is_holding | 公司标记 | 龙鱼库 holdings 单点 | 沿用现制 |

> 裁定依据：纯度/弹性是「品种↔公司」边上的量，塞进龙鱼公司 record 会逼出"一公司多品种数组 + 白泽弹性反向回写龙鱼"的反向依赖；留白泽侧最干净（Doctor 2026-06-29 拍板「留白泽侧·作涨价受益逻辑」）。

## 五、要弃用/简化的冗余（本案最实收益）

- **弃 `stock_scores` 表（226 行评分镜像）**：公司评分只在龙鱼库一处，结构上根除「226 镜像 vs 86 正本」漂移。
- **弃 `data/weekly/longyu_scores.json`**（白泽自存评分副本）+ `sync_longyu_universe.py` 的 (A) ingest 分支。
- 保留 `sync_longyu_universe.py` 的 (B) 扩池逻辑（白泽往受益关系/名单加新 ts_code），但下游改为「推龙鱼跑分」而非「写镜像」。
- 弃用一律 archived / 顶部加指针，**不删**（可逆优先）。

## 六、公共调用层（白泽 owner 之外，谁都能只读）

把烛照九阴现成的 `fundamentals_lookup.py`（已 `?mode=ro`）泛化成 `lib_public_read`，统一三个口：
- `lookup_company(ts_code)` → 龙鱼库（六维/评级/thesis/持仓）
- `lookup_benefit(ts_code | commodity)` → 白泽受益关系（纯度/弹性/品种）
- `lookup_commodity(commodity)` → 白泽品种行情
契约：龙鱼库＝公司公共库、白泽库＝大宗公共库，**各自 owner 写、跨项目只读 import**。

## 七、代价与缓解（不藏）

1. **owner 翻转**：原设想"白泽维护公共库"，实际变成"龙鱼是公司公共库（研究侧 owner）、白泽是大宗公共库"。需在两库 README/契约里写明边界。
2. **白泽→龙鱼 运行时读依赖**：白泽周报渲染要能读龙鱼库（在 `Database/` 内可达）。缓解：读失败时「龙鱼六维」列留空标注「待跑分」，不阻塞周报出稿（沿用占位禁入/缺口诚实）。

## 八、验收标准（落库后算 done）

- [ ] `stock_scores` 镜像 + `longyu_scores.json` 已 archived，白泽不再自存公司评分。
- [ ] 白泽周报「龙鱼六维」列改为实时调龙鱼库，数值与龙鱼库一致（抽 3 只核对）。
- [ ] 白泽留存的受益关系表字段＝§四白泽侧清单，无六维分冗余。
- [ ] 龙鱼库仍为公司唯一评分真源，PK 无重复、无张冠李戴。
- [ ] `lib_public_read` 三个口可用；两库 only-read 契约成文。
- [ ] 弃用项全部 archived + 指针，无物理删除。
