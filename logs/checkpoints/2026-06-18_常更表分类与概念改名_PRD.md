# PRD · 常更表分类机制 + 概念级改名 DVA-金融分析

## §一 · 任务目标

给 DVA 常更作者表（watchlist）引入 `category` 分类维度（金融 / 设计 / PEC语料），用一个交互界面让 Doctor 给每位作者"打勾"归类；`category` 与既有 `mode` 正交——category 决定**去向/消费者**，mode 决定**管线深浅**。同时把"DVA 后续分析模块/库"在**面向用户的称呼**层改名为"DVA-金融分析 / DVA-金融分析库"（不动任何物理路径、目录名与 DB root guard）。

**Doctor 原始指令**（逐字引用）:
> "常更作者表，帮我实现个"分类"，让我打勾。将DVA后续的分析模块改名为DVA-金融分析，相关库改名为DVA-金融分析库。
> 分类如下："金融"（下载-转写-DVA金融分析-入DVA金融分析库）；"设计"（有只下载的，有下载加转写的，都不分析，对接消费者：VV的Design Reference）；"PEC语料"（下载，转写，对接PEC）。"

**对齐结论摘要**（4 轮 AskUserQuestion 锁定）:
- **改名范围** = 概念/标签级：只改面向用户称呼（UI/widget label + dva.js help/usage + 使用流程.md + README.md）；代码目录名、变量、路径、`assertDvaDbPath` 硬编码一律不动。
- **打勾载体** = 交互 widget：Doctor 勾选 → 提交把分类结果回传 CC → CC 生成 `dva watch set-category` 命令清单 → Doctor 在 Mac 跑（artifact 碰不了本地文件，写回必须经 dva.js / Mac）。
- **category × mode 正交**：金融 ⇒ mode=full；PEC语料 ⇒ mode=transcribe-only；设计 ⇒ mode ∈ {download-only, transcribe-only}（逐人在打勾界面选），设计与 PEC语料**都不跑分析**。
- **消费者路由（仅标记，不建管线）**：金融 → DVA-金融分析库；设计 → VV 的 Design Reference；PEC语料 → PEC。
- **MVP 边界**：只做 category 标记 + mode 路由 + 打勾 UI；实际投递（把字幕/视频送到各消费者）沿用现有产物目录，本次不建投递管线。

**任务规模估算**:
- 预计涉及文件: `dva.js`、`基础模块/watchlist.js`、`使用流程.md`、`README.md`（+ 执行时生成的打勾 widget），约 4 改 1 新
- 预计耗时: 1–2 小时
- 涉及项目: DVA（仅 `Projects/DVA/` 代码仓 + brain/DVA 文档）
