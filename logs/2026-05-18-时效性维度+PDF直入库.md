## 做出的决策

| 决策 | 原因 | 影响 |
|------|------|------|
| 外资 PDF 直接 batch（不进 Notion 转 md）| Notion 转 md 是手动步骤、外资研报量大；V4 Pro 多模态可直读 PDF | 节省 Doctor 大量手动操作 |
| 防御性补登 66 个旧 PDF 名到 index.json | 之前通过 md 路径 ingest 但 index 只记 md 名；防止 batch PDF 模式重复 ingest | 节省 ~1 小时 LLM 调用成本 |
| `status="expired"` 节点保留在 canonical 但默认不可视化（1A）| 保留历史信息可追溯，但默认视图不污染 | --include-expired CLI 可调试 |
| 信号 ever 状态加"曾·"前缀（2A）| 比新桶视觉清晰；前端 signal_categories 清空即自动失效特效 | 无需改 starscape-template.html |
| 自动 quarterly_recheck（3A）| Doctor 不用手动维护每季度复检 | kg_ingest 跑完即识别过期 |
| 英文短词用 word boundary 而非 substring | 避免"exportPolicy"含"tP"误命中"TP" | 误判率从 213/348 → 144/278 (-32%) |
| ever_continuing 分支必须显式 `status="ever"` | v1.0 没设回，导致下次复检 `old_ever=False` 走 ever_new 误触发 | name 双重防御 + status 保持 |

## 工程沉淀

- **PDF 多模态 batch 比 Notion 转 md 工作流省 ~10 倍人力**（37 个 PDF 自动跑 vs 37 次手动操作）—— 外资研报路径以后默认走 PDF 直入库
- **状态机的"保持转移"必须显式**：compute_status 算出 expired，但 DB 状态可能要保持 ever；不要假设默认覆盖会自动同步
- **英文短关键词在中文环境必须 \b boundary**：substring match 会让短英文词在长英文 id（如 "exportPolicy"）里大量误命中
- **防御性"双轨"判定**：already_ever = (old_status=="ever" OR name.startswith("曾·"))，比单一字段判断稳得多
- **自动 hook 链**（batch → quarterly_recheck）大幅减少维护成本，但要小心数据一致性边界
