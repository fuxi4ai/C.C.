## 做出的决策

| 决策 | 原因 | 影响 |
|------|------|------|
| 定时任务架构走方案 B（Codex 主跑 + Claude 消费握手） | 能力分层最优：重活给 GPT-5.5、细活留 Claude；规矩执行留主场；握手 JSON 可审计；不动 Claude 日常对话 | 所有定时任务需逐个迁移到 Codex 侧；Claude 侧起 handshake-consumer skill + scheduled task |
| 握手目录固定为 `~/Documents/4AI/Shake hands/{to CC, from CC}/` | Doctor 命名，走本地不走 iCloud；`to CC`=Codex→Claude 方向、`from CC`=Claude 回执方向 | 目录约定进 README，做为跨系统契约 |
| 握手文件保留策略：最新 + 最近一份副本（共 2 份） | Doctor 追加约定——最新用于消费、一份备查便于回溯诊断；只留一个副本、不无限堆积 | Codex 侧覆盖 `{task_id}.latest.json` 前先把当前 latest 挪到 `{task_id}.prev.json`（滚动一位）；Claude 侧只读 latest |
| 握手 payload 只装数据 + 建议动作，不夹带 Codex 自作主张的落盘 | 让"数据真实性铁律""promote 需 Doctor 显式"等规矩全部留在 Claude 侧执行，Codex 不越权改 canonical | Codex 端 prompt 保持简单；schema_version + checksum 兜住版本漂移 |
| 触发方式先走"Claude 保留一个薄定时任务每天 09:00 醒一次"消费握手 | 最省心、成本最低（consumer 任务本身很轻）、Doctor 无需记流程 | 后续若量级变大再考虑 launchd/fswatch |
| 迁移策略：先起骨架（目录+schema+README+consumer skill），再拿投知君君这一个真任务走通端到端，再批量迁 | 单点验证优先、避免大爆炸迁移 | 换 opus4.8 新对话动手 |
