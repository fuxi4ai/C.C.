## 工程沉淀

- **review_hold 处理方法论**：按"结构性原因"分类（节点错配/跨域类比/含糊）而非按 type；不同类用不同手段（新增节点 / 修指向 / 改 type / 删边）
- **新增节点优于删边**：当节点错配明确（target 应是 metric 而非公司），新建 metric_* 节点保留产业语义比删边好
- **删除标准**：跨行业纯类比 + 间接到几乎无关 → 删；含糊但有产业链关联 → 改 type；schema 缺桶但语义有价值 → 改最接近的 type（如"问题 about 材料" → part_of）
- **审计字段约定**：`_meta.review_hold_resolved_YYYY_MM_DD` 记录 action / 原值 / 新值 / reason，可 grep 回溯
- **CLI 工具设计**：`kb.py` 检索 vs `signal_impact_chain.py` 分析，两者职责分离；signal_impact_chain 的"跨信号汇总"输出（找被多个信号影响的"脆弱下游"）是产业链脆弱性分析的杀手锏
