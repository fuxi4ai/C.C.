## 完成的工作
- 承接四批外移（1556 条）已闭环，推进删源前置纪律「抽样恢复演练 + 逐批批准」
- 建 `recovery_drill/` 工具集：抽样清单（30条分层，覆盖4批+大小极值+最长路径）、`verify_recovery_ssh.py`（ssh 远端就地哈希）、`delete_source.py`（三闸删源）、四批全量清单 `full_b0x.json`
- 抽样恢复演练 30 条全 PASS（0.6MB~478MB），远端字节完整 + ssh 取回通道确认可用
- 逐批全量远端核验：b04 208/b03 612/b02 298/b01 438 全 PASS FAIL 0
- 三闸删源（远端已核验 + 删前本地哈希复核 + 默认 dry-run），只删 mp4 保元数据：四批 receipt 全部 `source_deleted=true`
- 收官：Downloaded mp4 从 1556 → 0，腾出 ~71.5GB；元数据（1572 json + 93 webp，~0.16GB）原样保留
- 更正前次误判：先前说「3 条 path_exists=false 无源」实为沙箱路径映射假阴性——DB 存 `/Users/...` 真路径，沙箱挂在 `/sessions/.../mnt/`，用原路径判 exists 必 False；真相是「文件夹在、mp4 不在」

## 做出的决策
- 删源前置关：恢复演练证「取得回」而非只「记录在」——receipt 的 pointers/db_verified 只证上传写对，不等于远端可完整取回，故补 ssh 全量远端哈希核验这道独立闸
- 删源三道闸：① 远端全量核验（fuxi 有）② 删前本地哈希==pointer记录（本地是同一份、防删错）③ 默认 dry-run，--commit 才真删
- 删源粒度只删 mp4：空间 99.8% 在 mp4（71.47GB vs 元数据 0.16GB），保留 `_data.json`/`.webp` 唯一原件，可逆友好
- 删源状态记 receipt.source_deleted（体系内建字段），不给 DB 加列（不擅改 schema）
- ssh 校验 subprocess 收 bytes 自解码（utf-8→gbk 兜底）：fuxi 中文 Windows 的 stderr 是 GBK，text=True 会 UnicodeDecodeError 崩
