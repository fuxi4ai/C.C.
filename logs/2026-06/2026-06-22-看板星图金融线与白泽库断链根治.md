### 线二 · 白泽大宗-商品库断链诊断 → 根治
- **定位真因**：看板断链=snapshot 以 `mode=ro` 读 `business_breakdown.db` 报 `attempt to write a readonly database`。根因是残留 hot `-journal`（6/21 周更写库被挂载盘 I/O 打断、事务未提交）→ sqlite 打开必须回滚而回滚要写→只读挡住。沙箱可写副本验证：回放干净、integrity ok。
- **应急补跑**（Doctor 真机）：备份→`integrity_check` 清热日志→`build_warehouse.py` 补跑 Stage 5 ingest→6 表入库、ingest_meta 推到 06-22、库恢复只读可读、不再断链。
- **根治**：`build_warehouse.py` 加挂载盘直写兜底——撞 disk I/O/readonly 自动「拷 /tmp→副本 ingest（顺带回滚 journal）→残留 journal 截断为 0 字节→/tmp 整库 cp 覆盖回写」。沙箱真实挂载库实测兜底触发+回写成功、幂等不误伤既有表。沙箱单轨从此自落库。
- **清 git 陈旧锁**：白泽大宗 + DVA 两仓 `.git/index.lock`（6/16 陈旧空锁）卡 commit → 真机 `rm -f` 解。
- **提交**：白泽大宗仓 commit build_warehouse 根治 + 本周数据真源(5 JSON)+周报 + GOTCHAS。
- **落 GOTCHAS**：更新 GOTCHA-027 状态为「已根治」；新增 GOTCHA-028（hot journal→只读 readonly→看板断链·应急复原+自动回写+0字节journal+index.lock）；更新日志加 06-22 行。
