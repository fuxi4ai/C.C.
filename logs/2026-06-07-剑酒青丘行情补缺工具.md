## 完成的工作

- **加补缺规则**（`docs/INDEX 关键约定` + `GOTCHAS G04`）：行情库缺失可调 tushare pro 补，四守则——① token 走 `Database/.env` 的 `TUSHARE_TOKEN`（`config.load_env`），不用 .token.env.b64/不硬编码/不入库；② CC 不在 sandbox 抓（句芒 owns 取数），Doctor 在 Mac 跑或并入句芒日更；③ 写回公共 `market_data.db`（stock_daily 列对齐）；④ 缺口检测可沙箱纯 Python，fetch 在 Mac。
- **建 `gap_detect.py`**（沙箱安全·零网络）：按交易日历（daily_market 1552 天）扫 stock_daily，分"中段缺口"(288股)与"断尾"(2股)，输出待补清单 JSON。实跑验证。
- **建 `fill_gaps.py`**（联网·仅 Mac）：读清单→tushare daily 按区间拉→`INSERT OR IGNORE` 写回（PK=ts_code,trade_date 天然去重）。dry-run 验证区间计算正确。
- **Doctor 实跑确认库完整**：tushare 返回 15801 行 / 新增 0 行 → 那 288 个"缺口"确为停牌日，非数据漏洞。补缺工具验证有效（能拉、能去重）。
- **诊断增强**：原 `cur.rowcount`(executemany 下不可靠)→改 `con.total_changes` 精确计；并单列"tushare 返回行数"以区分"源没返回(token/权限/网络)"vs"返回但全已有(库完整)"。

## 做出的决策

| 决策 | 原因 | 影响 |
|------|------|------|
| 补缺＝检测(沙箱) + 补数(Mac) 两段分离 | CC 不在沙箱联网抓数；token 安全 | 检测零网络出清单，fetch 在 Mac 用 env token |
| 旧取数脚本(data_manager/fetch_stocks/query_tushare_pro)不复用 | 硬编码 openclaw 死路径 + .token.env.b64 不合规 | 新建干净 gap_detect/fill_gaps；旧的留 G04 标注 |
| 中段缺口默认不强补、只列候选 | 多为停牌日(tushare 也无) | fill 拿清单问 tushare，停牌日自然补不回(自我纠正) |
| 补数计数用 con.total_changes + fetched 双计 | executemany 的 rowcount 不可靠、且要区分空源vs已满 | 一眼判断 token 问题还是库已完整 |
