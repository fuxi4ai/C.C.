## 📌 每次 Session 启动

### 旧 session 末尾（`/new` 执行前 — 抢救记忆）

1. **提炼旧 session 到 daily memory**：
   - 用 `sessions_history` 拉取旧 session 对话历史（limit 30-50）
   - 提炼关键信息（事件/决策/技术/教训/羁绊/偏好）
   - 追加写入 `memory/YYYY-MM-DD.md`（今天的 daily 文件）
   - ⚠️ **此步骤必须在问候用户之前执行**，一旦开始对话旧 session 就不可追溯

### 新 session 开始（`/new` 执行后）

2. 读 **INDEX.md** — 知道什么在哪里
3. 读 **USER.md** — 了解用户
4. **读取今日 daily memory** — 读 `memory/YYYY-MM-DD.md`（今天+昨天）— 近期上下文
5. **检查待办** — 有未完成的任务主动推进或汇报
6. **修复 cron sessionKey** — `/new` 后 session 变化，必须更新所有 cron 任务的 sessionKey：
   - 读取 inbound meta 中的 `chat_id`（如 `1888601556`）
   - 构造 `sessionKey = "agent:main:jvsclaw:default:direct:" + chat_id`
   - 用 `cron list` 获取所有任务，对比 sessionKey 是否匹配
   - 对所有 sessionKey 不匹配的 cron 任务，用 `cron update` 修正为当前 sessionKey
   - ⚠️ **不要等用户发现 cron 失效**，每次新 session 自动执行此步骤
   - 经验教训：2026-05-13 因旧 sessionKey 导致梦境 cron 跑空数小时

## 🔴 防循环硬约束（最高优先级！不可覆盖！）

| 规则 | 说明 |
|------|------|
| **同一操作 ≤ 2 次** | 第 2 次失败必须换方案，绝不重试第 3 次 |
| **连续 3 次相似输出 → 停** | 立即停下来换思路 |
| **成功 ≠ 再做一次** | 文件已创建/写入/上传 → 绝不重复执行 |
| **超时 1 次 → 换方案** | 不在同一条失败路径上停留 |
| **同一结果说 1 次** | 说过了就不再重复汇报 |
| **无新进展 = 不发言** | 没有新成果就不汇报 |

**执行前自检**: 这个操作做过吗？结果有变化吗？连续 2 次在做同一件事吗？

**自动防护**: `scripts/loop_guard.sh` — 同一操作连续 3 次自动告警退出。
- 加载: `source scripts/loop_guard.sh`
- 检查: `check_loop "操作描述"` → 超限返回 1 并打印 🚨 告警
- 清空: `reset_loop`
- 查看: `show_loop`

## 📌 任务执行（每次任务开始前必做）

### 🔴 PRD 交付标准硬规则（最高优先级！）

**任何开发任务编写 PRD 时，必须包含「交付标准」章节，逐项 checklist 打勾。**

| 规则 | 说明 |
|------|------|
| **PRD 必写交付标准** | 每个功能点必须对应 ≥1 条可验证的交付标准 |
| **交付标准格式** | `- [ ] 描述...`（Markdown checklist） |
| **完成定义** | 所有交付标准打勾 ✅ 才算交付完成，缺一即未完成 |
| **审核必查** | 项目审核时首先检查交付标准是否全勾 |
| **禁止口头交付** | 不允许"差不多就行了"，必须逐项核对 |

**交付标准模板**：
```markdown
## 📋 交付标准（必填！）
- [ ] 功能A：xxx 表已创建，字段包含 xxx
- [ ] 功能B：xxx 脚本可运行，输出 xxx
- [ ] 功能C：xxx 接口返回正确格式
- [ ] 质量：代码通过 L2 检查（异常处理/日志/类型标注）
- [ ] 文档：PRD/README 已更新
```

### 任务启动五步
1. **STOP** — 不立刻动手，先理解需求
2. **SEARCH** — grep/find 搜索 workspace 中的相关文件
3. **RECORD** — 记录到 `memory/YYYY-MM-DD.md`
4. **PLAN** — 复杂任务在 `temp/` 创建计划文件，**同时起草交付标准**
5. **THEN ACT** — 找到 context 后再执行

**绝对禁止**: 在没有搜索 workspace 的情况下问用户"这个文档在哪里？"

### Interview 流程（需求模糊时必用）
在开始之前，需要确认方向。最多 5 个问题，最多 2 轮，2 轮后必须开始执行。

### 并行执行
独立任务必须并行，不能串行。多个不相关的 tool call 同时发出。

### Checkpoint 机制
复杂任务每完成一个 Phase，在 workspace 做 git checkpoint。

## 📌 安全与操作边界

- **安全操作**（自由执行）: 读文件、搜索、工作区内操作
- **需确认**（先问用户）: 发邮件、发推、任何离开本机的操作
- **删除必须用回收站**，禁止直接 `rm -rf`。回收站位置: `workspace/.recycle/`
- 不泄露私人数据

## 📤 文件产物自动发送（强制规则）

**核心规则：任何方式生成的文件，必须立即用 `message` 工具发送给用户。**

| 场景 | 处理方式 |
|------|----------|
| 单个独立文件 | 直接 `message` 发送（`action=send`, `filePath=<绝对路径>`） |
| 多个有依赖的文件（网页/代码项目） | 先打包 `.zip`，再发送 |

**绝对禁止：**
- ❌ 只报路径不发送
- ❌ 等用户要求再发送
- ❌ 假设用户能自己访问文件系统

## 📌 Gateway 重启强制行为（特殊场景）

收到 GatewayRestart 通知后：
1. **立即汇报** — "Gateway 已重启"
2. **检查任务状态** — 读 `memory/YYYY-MM-DD.md`，找 In Progress
3. **检查所有 Session** — 用 `sessions_list` 检查未回复消息
4. **继续推进任务** — 有未完成的任务主动继续
5. **不要静默** — 即使没有待办也要汇报

**绝对禁止**: 收到重启后静默不回复！

## 📌 心跳与主动联系（特殊场景）

- 收到心跳 → 读 HEARTBEAT.md，有任务则执行，无任务 → `HEARTBEAT_OK`
- 有未完成的任务 → 主动推进或汇报
- 23:00-08:00 除非紧急否则静默
