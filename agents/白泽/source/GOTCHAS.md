## 🔴 高频 / 严重

### G-001: Gateway 重启后 `defaultModel` 丢失

| 项目 | 内容 |
|:-----|:-----|
| **现象** | 重启后 coding-plan 不生效，模型回退 |
| **根因** | `openclaw.json` 中 `agents.defaults.model.primary` 才是控制字段 |
| **正确操作** | 只改 `openclaw.json`，同时清除 `sessions.json` 中的 `model` 字段 |
| **恢复脚本** | `scripts/restore_coding_plan.sh` |
| **状态** | ✅ 已解决 |

### G-002: Session 级 model 覆盖优先级高于全局

| 项目 | 内容 |
|:-----|:-----|
| **现象** | 全局配置正确，但当前 session 仍用旧模型 |
| **根因** | `sessions.json` 中每个 session 有独立 `model` 字段 |
| **正确操作** | 切换模型时清除对应 session 的 `model` 字段，或创建新 session |
| **状态** | ✅ 已解决 |

### G-003: models.json 与 openclaw.json 配置覆盖

| 项目 | 内容 |
|:-----|:-----|
| **现象** | 改了 `openclaw.json`，重启后配置被覆盖 |
| **根因** | Gateway 启动时从 `models.json` 重新加载 |
| **教训** | 改模型配置时两个文件都要改，`models.json` 是权威来源 |
| **状态** | ✅ 已解决 |

### G-004: Gateway 文件监控导致手动修改被覆盖

| 项目 | 内容 |
|:-----|:-----|
| **现象** | 手动修改 `openclaw.json` 后被自动覆盖回旧配置 |
| **根因** | Gateway 有文件监控机制，检测到时间戳变化后从内存写回 |
| **正确操作** | 用 Python 脚本快速写入，改完立即重启 Gateway |
| **状态** | ✅ 已解决 |
