## 处置 / 坑（⚠️ 挂载盘大库写入·今日踩实一个）
- **cp 直接覆盖已存在的大库 → FUSE 挂载盘会「回滚」，写不进去**。今日首次直接 `sqlite3.connect(挂载盘db)` commit 撞 `disk I/O error`（已知）；退回 `/tmp` 副本往返后，用 **`cp 副本 → 挂载盘同名`** 放回，`md5` 一度匹配但随即回退到旧文件（size/md5 双双变回原值，读出 stock_daily 竟停在旧值）。
- **解法（今日验证）**：放回**不能** cp 覆盖原文件，必须 **写新名 + 原子 rename**——`cp 副本 → 挂载盘/market_data.db.new`（新文件写入稳定、md5 三读一致），再 `mv -f *.new market_data.db` 原子替换。放回后 `sync + sleep`，`immutable=1` 读取校验 max=20260722、quick_check=ok。此即昨日 log 里「cp→mv」范式的所以然，今日直面并坐实：**关键是 mv 原子替换，绝不 cp 直接盖原文件**。已同步回写 [[通用教训]] GOTCHAS。
