## 2026-05-16 · 关系类型扩展至 10 种（新增 causes / constrains）

**背景**：V4 Pro 在 18 篇研报批量入库时自发输出了 5 条规范外关系（2 条 causes、3 条 constrains），不在 v2 原定 8 种类型集合中。这些边语义合理：
- causes：事件传导/状态变化引发的强因果（如"服务器量缩 → CPU 恐慌囤货推高存储价格"）
- constrains：硬约束/限制/拖累（如"三星代工产能瓶颈 → 海光 CPU 7000 系列"）

**选项**：
- A：接受扩成 10 种类型，更新 prompt + rel_classify + 文档
- B：跑 `kg_rel_classify.py` 归并到 8 种（causes → enables、constrains → requires）

**选择**：A

**理由**：
1. enables 已经吃了 solved_by/trades_off/scaffolds，再吞 causes 会让该桶过度泛化、丧失分析价值
2. causes/constrains 是产业链分析的高价值关系（事件传导、政策反应、价格联动、产能瓶颈），归并必丢信息
3. kg_rel_classify.py 自身已有 constrains 桶，schema 演化方向就是更细分类型，prompt 那 8 种是保守起点而非终态
4. enables 表达"能力解锁（持续）"，causes 表达"事件传导（一次性）"，正交清晰

**影响**：
- `kg_ingest.py` prompt 扩到 10 种 + 加判断口诀
- `kg_rel_classify.py` 加 causes 桶（在 constrains 前）
- 系统概览.md / CLAUDE.md 数字与类型种数同步
- 后续入库的研报会主动使用 causes/constrains，旧 canonical 不回填（数据增量自然演化）

## 2026-05-16 · Schema v3 重设计 · 19 种归并到 11 种核心 schema

**背景**：图谱存在双字段 schema 不一致问题——`type` 字段（V4 Pro prompt 阶段写，10 种）与 `relation` 字段（kg_rel_classify.py 历史输出，19 种）并行存在，1155/1666 条边同时有两个字段值不一致；wiki_autogen.py 读 `relation`，kb.py 读 `type`，前后端看到不同 schema。

**问题诊断**：
- `relation` 字段（19 种旧 schema）里 466/1155 是 `relates_to` fallback，47% 边语义未明确
- prompt 强制 10 种把 supplies/manufactures 类边压成 part_of，丢失"产业链供应关系"分析维度
- 反向边对（buys_from↔supplies、evolves_to↔evolves_from 等）从未做 source/target 归一化

**Doctor 设计原则**（2026-05-16）：
1. 可以删字段、改源码
2. 尽量保留语义和逻辑（分析的正确性最重要）
3. 相同语义和逻辑的尽量并归

**选项**：
- A：保留 10 种现状（最低风险，损失 supplies 维度）
- B：扩到 11 种 + 完整迁移（新增 supplies，反向边归一化，删 relation 字段）
- C：扩到 19+ 种（推翻规范化目标，schema 持续膨胀）

**选择**：B

**11 种核心 schema**（按模态逻辑分类）：

| # | 类型 | 模态 | 语义 |
|---|------|------|------|
| 1 | causes        | 充分条件        | 事件传导/触发 |
| 2 | enables       | 必要条件（上游） | 能力解锁/持续支撑 |
| 3 | requires      | 必要条件（下游） | 依赖/前置（enables 反向对偶）|
| 4 | constrains    | 负向必要        | 硬约束/限制 |
| 5 | supplies      | 产业链结构      | 上游对下游的供应/代工/制造/设计服务 |
| 6 | part_of       | 产业链结构      | 仅内部构成（组件） |
| 7 | used_in       | 产业链结构      | 被用于/应用场景 |
| 8 | evolves_from  | 时序            | 演化/迭代/替代/派生 |
| 9 | competes_with | 横向            | 竞争 |
| 10| is_a          | 分类            | 类属 |
| 11| measured_by   | 度量            | 指标关联 |

**归并表**（19 → 11）：

| 旧 schema | 新 type | 处理 |
|---|---|---|
| component_of           | part_of       | 真同义 |
| design_service_by, manufactures | supplies | 上游身份特化 |
| drives, impacts_cost   | causes        | 真同义 |
| supports, solves, optimized_for | enables | 同义 + 边界归并（solves/optimized_for 损失部分语义）|
| replaces, derived_from | evolves_from  | 演化语境 |
| benefits_from          | requires      | 受益≈依赖 |
| tracks                 | measured_by   | 真同义 |
| buys_from              | supplies      | **反向边**：swap source/target |
| evolves_to             | evolves_from  | **反向边**：swap |
| constrained_by         | constrains    | **反向边**：swap |
| manufactured_on        | used_in       | **反向边**：swap |
| relates_to (240+466)   | 保留 type     | fallback，用 prompt 阶段的 type 兜底 |

**逻辑结构（充分/必要矩阵）**：

```
              正向（让 B 出现/增强）   负向（让 B 不出现/受限）
   充分条件   causes (事件传导)        [prevents — 暂无]
   必要条件   enables / requires*      constrains
              (反向边对偶)
```
* enables 与 requires 互为反向边等价；按研报叙事方向选用，不双向重复建边

**影响**：
- `kg_ingest.py` prompt 重写为 11 种 + 完整语义定义 + 判断口诀
- `kg_rel_classify.py` RULES 重写（19→11，supplies 加新规则）
- `wiki_autogen.py:130,145` 改读 `type` 字段（兼容 fallback `relation`）
- 迁移脚本：1666 条边 type 字段统一到 11 种，95 条反向边交换 source/target，1155 条 relation 字段删除
- 产物：`mapping/行业知识图谱_v3_20260516_152824_schema11.json`（待提升为 canonical）
- 系统概览.md / CLAUDE.md 同步数字与 schema 表

**迁移统计**：
- type 字段被修改: 470 条（其中 95 条同时 swap）
- 反向边交换 source/target: 95 条
- relation 字段删除: 1155 条
- 规范外 type: 0 ✓

## 2026-05-17 · Schema v3 落地完成 · 全图谱重打标签

**背景**：2026-05-16 决策的 11 种 schema 重设计，初次迁移用 rel_classify regex + type/relation 双字段归并的方案，验证发现 73 条 evolves_to swap 方向错乱、rel_classify 规则覆盖率不足等问题（v3 中间产物已隔离至 `mapping/_quarantine_v3_*_aborted.json`）。

**最终方案**：B+ 增强版 · 由 V4 Pro 在新 prompt 下对全图谱 1650 条边逐条重判 type + source/target 方向。

**实施关键**：
- **prompt 改造**：从"swap_source_target boolean"换成"correct_source/correct_target 双字段直输"，避免 LLM 做"反转推理"误差（pilot 第一轮 swap 0/3 准确率 → 第二轮 3/3 ✓）
- **置信度阈值**：从 0.7 调到 0.85，匹配 V4 Pro 实际置信度分布（均值 0.918）
- **review_needed 机制**：短描述（<15 字）+ 低置信（<0.85）+ 方向警告，三类自动标记入 `_meta` 字段
- **手工兜底**：V4 Pro 偶发的"自创 type"（如 `uses`）由脚本 `continue` 跳过；事后人工补 2 条 BroadcomXPU 边

**结果**：

|------|------|
| 处理边数 | 1650 / 1666（16 条短描述自动标 review） |
| type 改写 | 678 条（41%）|

| 置信度均值 | 0.918 |
| 失败批 | 0 |
| 耗时 | 80 分钟（DeepSeek V4 Pro · thinking disabled）|
| review_needed 总数 | 132 条（7.9%）|

## 2026-05-17 · Schema v3 落地后 · 3 轮联合审计 132 条 review_needed

**背景**：B+ 增强版全量跑完后，132 条边被标 `_meta.review_needed`（116 低置信 + 16 短描述）。按 V4 Pro 决策修改组合做 3 轮分批人工审计 + 一键 apply 修复。

**审计方法**：

2. **Round 2（type改+未swap 36 条）**——单维度修改，错误率中等（64% 准确）

| 轮次 | 范围 | 修 | HOLD | 接受 | V4 Pro 准确率 |

| Round 1 | type改+swap 14 | 6 | 0 | 8 | 57% |

| **累计** | **132 条** | **35** | **15** | **82** | **70%** |

**主要修复方向**（35 条修复中的高频模式）：
- enables → supplies（多）：V4 Pro 把产业链供应关系（代工/设计服务/向...供货）压成"使能"
- enables → causes（多）：把因果驱动（"推动"、"拉动"、"驱动"）压成"使能"
- part_of → supplies（多）：把外部公司供应关系误归"内部组成"
- measured_by → part_of/requires/causes（少）：target 是 metric 时 V4 Pro 倾向 measured_by，但 description 主导关系含义
- evolves_from + swap：方向反转（同代不同档次产品被误判为代际派生）

**15 条 review_hold 留待节点重构**（结构性问题）：
- **节点错配**（7 条）：target 应是 metric 而非公司/产品；source 用错节点；desc 主语跟 source 不匹配
- **跨域类比**（5 条）：跨行业类比关系（如光伏玻璃 vs 水泥供给侧），schema 11 无合适桶
- **关系含糊**（3 条）：业务模式/缺席性竞争/不同部位组件等，需 schema 扩展或重设计 edge
