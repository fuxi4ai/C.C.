# brain-prd — 多轮对齐 + 在场起草 PRD（go-creator）

> **来源**：[[视频方法论对照与开发流程优化-设计提案]] §3.1 / §4(B)，Doctor 2026-06-13 裁定建独立 skill；2026-06-13 二轮反馈：**起手的多轮对齐需详细设计**（借鉴视频"毒舌产品经理"逼问需求）。
> **两段价值**：① **对齐**——开工前把核心需求/方向/验收和 Doctor 问清楚，不靠 CC 猜；② **起草**——对齐后 CC 在场把整份 PRD（尤其最费神的可验证交付标准）写好，你只审/改/批。
> **不改 G-X4 任何治理铁律**：三态评估、Doctor 独占打 ✓、双向闭环关闭、沙箱禁 git 一律照旧。

---

## §1 · 何时用（When）

**用**：
- 显式：`/prd`、`/prd [任务简称]`、"立PRD"、"起草PRD"、"写交付标准"、"写个 go"。
- 主动：CC 判断任务**规模超阈值**（多文件改 / 跨小时 / 跨项目 / 凌晨自主 / Doctor 不全程监督）→ **主动提议**"建议立 PRD，先和你对齐再起草"，Doctor 同意才进。

**不用**（避免官僚化）：单一问答 / 解释 / 研究讨论；单文件小改（< 10 处）；Doctor 说"不用立"。

---

## §2 · 什么算合格（Done · 可验证）

1. **对齐充分**：核心需求 / 开发方向 / 交付验收 三维都已问清或 Doctor 明确"无疑问"，且 CC 已**复述对齐结论并得到 Doctor 确认**（见 §4 Step 2）。
2. **文件已落**：`~/Documents/Claude/brain/logs/checkpoints/{YYYY-MM-DD}_{任务简称}_PRD.md` 存在（`ls` 可见），从 `templates/prd-acceptance-template.md` 复制填成。
3. **§一 目标**：含动机 + 范围 + **Doctor 原始指令逐字引用** + **对齐结论摘要** + 规模估算。
4. **§二 交付标准全部可机器验证**：每条形如"文件存在于 X / `Grep 'Y' = 0 matches` / 脚本输出 Z"；§二全留 `[ ]`（起草阶段不填三态）。**Grep 自检：§二 内无"做好了/完整/修复了/优化了"等主观词。**
5. **§三 非交付项**：写明本任务不含什么（用对齐时划定的边界）。
6. **frontmatter `doctor_decision: 待`** + 回报点明"这是起草稿待你审/改，CC 不会自行开工/打 ✓"。

---

## §3 · 什么红线（Never）

- ⛔ **对齐不足不硬起草**：三维有硬缺口（尤其"到底要解决什么问题""怎么算验收"）未问清，**不许**跳到起草。
- ⛔ **禁阿谀奉承**（毒舌纪律）：不附和、不夸"好主意"、不顺着含糊需求往下滑。答案模糊 / 自相矛盾 / 范围膨胀 → **当场指出 + 要明确**。但毒舌＝较真，不是无礼，守 [[Doctor协作偏好]] 的简洁与尊重。
- ⛔ **不一次轰炸**：一次聚焦一个维度 / 1–2 个问题，优先选择题，别甩一长串问卷。
- ⛔ **CC 永远不打 `[✓]`**：只有 Doctor 能 `[?]`→`[✓]`。起草阶段 §二 全留 `[ ]`。看到自己将打 `[✓]` → 立即停。
- ⛔ **不自动关闭**：只"全 ✓"或"Doctor 显式取消"两条合法路径；不得 commit 即关 / 标完 Task 即关 / "我觉得完了"即关。
- ⛔ **交付标准不许主观陈述**：不可验证 = 等于没写（G-X4「可验证=客观」）。
- ⛔ **不在沙箱跑 git 写命令**（G-X2）：落盘走 Write；要 commit 贴命令给 Doctor。
- ⛔ **起草 ≠ 开工**：交出起草稿后停下等 Doctor 审/改/批（propose-then-confirm）。

---

## §4 · 执行步骤

### Step 1 · 判定要不要立
按 §1。主动场景先问一句"建议立 PRD，先对齐再起草，可以吗？"，同意再继续。

### Step 2 · 多轮对齐（★ 核心 · 起草前必做）

目标：把"模糊想法"逼问成"清晰可起草的需求"。流程四小步：

**2a · 先复述理解，校准方向**
CC 用"在场"优势，先一句话复述对任务的理解（要解决什么、大概怎么做），交 Doctor **确认或纠偏**——避免后续提问无的放矢。

**2b · 分轮提问（三维度 · 用 §5 问题库）**
按"核心需求 → 开发方向 → 交付验收"顺序，**一次聚焦一个维度、1–2 个问题**，**优先用 AskUserQuestion 工具出选择题**（带推荐项 + trade-off 说明），关键处配简答。一维问清再进下一维；Doctor 答"这维没疑问/你定"即可跳过该维细问。

**2c · 毒舌纪律（边问边卡）**
对照 §5 的「打回信号」：答案含糊（"优化一下""弄好看点"）、自相矛盾、范围膨胀（"顺便把 Y 也做了"）、方案冒充需求（跳过问题直接给做法）、忽略已知环境硬约束 → **当场指出 + 给选项让 Doctor 收敛**，不附和。

**2d · 对齐收口（进起草的闸）**
三维够了，CC 复述**对齐结论**：
```
对齐确认（请点头或纠正）：
- 核心需求（一句话）：…
- 开发方向 / 最小可交付：…
- 验收口径（硬标准几条）：…
- 非交付项（不做什么）：…
- 关键约束 / 红线：…
```
Doctor 确认 → 进 Step 3。**未确认不起草。**

### Step 3 · 在场补采技术细节
基于对齐结论，扫上下文把可验证标准落到实处：会动哪些文件/目录（→文件层面）、改完哪些旧名不该残留（→`Grep = 0 matches`）、哪些产物要能跑通/打开（→功能层面）；读所属项目 `architecture/系统概览.md` + `GOTCHAS.md` 取项目专属红线 → 写进 §二F / §三。

### Step 4 · 复制模板填写
读 `~/Documents/Claude/brain/templates/prd-acceptance-template.md`，复制到
`~/Documents/Claude/brain/logs/checkpoints/{YYYY-MM-DD}_{任务简称}_PRD.md`。
填 §一（动机 + Doctor 原话 + **对齐结论摘要** + 规模估算）、§二（A 文件 / B 一致性 / C 功能 / D 自审 / E 沟通 / F 任务专属，**全可验证、全留 `[ ]`**）、§三（非交付项，来自对齐）。

### Step 5 · 自检（落盘前）
- `Grep` §二 是否混入主观词 → 改成可验证形式。
- 每条问"能用一行命令判真假吗？"否 → 重写。
- §二/§三 是否忠实对齐结论（没擅自加项 / 没漏 Doctor 在意的验收点）。
- 涉 git 则沟通层面含"贴命令给 Doctor"（G-X2）。

### Step 6 · 回报（交起草稿）
```
📝 PRD 起草稿已落（已按对齐结论起草，待你审/改/批，我先不开工）
📁 logs/checkpoints/{YYYY-MM-DD}_{任务简称}_PRD.md
   核心需求：{对齐确认的一句话}
   交付标准：{N} 条（全可验证，已留 [ ] 待执行时填三态）
   非交付项：{M} 条
👉 ① 改/加标准 → 我修；② 说"开工" → 我执行并逐条填 [?]+证据；③ 说"取消" → 归档。
   注：✓ 只你打，我全程不打 ✓、不自动关闭。
```

### Step 7 · 移交执行（Doctor 批准后）
Doctor "开工" → 按 PRD 执行，**每达成一条立即** `[ ]`→`[?]`+证据；达不到 `[!]`+原因；不确定 `[~]`。结束改 §四"待 Doctor 验收"，触发 /save 贴 PRD 路径。**仍不打 ✓**。

## §5 · 对齐问题库（参考 · 三维 × 要问/算清/打回）

> 这是 what 不是 how，可写厚（遵 [[skill写作规范]]）。按维度取用，不必逐条问。

### 维度一 · 核心需求（要解决什么、为谁、为什么现在）
- **要问**：这次到底要解决什么问题？为谁？成功长什么样（一句话可感知）？不做会怎样（定优先级）？
- **算清**：能用一句话说出"痛点 + 谁受益 + 可感知的成功态"。
- **打回信号**：目标含糊（"优化/弄好看/完善一下"）；能解释任意结果的万能目标；**把方案当需求**（直接说"做个 X"却说不出 X 要解决的问题）。

### 维度二 · 开发方向 / 形态（范围、做法、约束）
- **要问**：范围多大？**最小可交付**是什么？有几种做法、各 trade-off、倾向哪种？改存量还是新建、波及哪些已有文件/项目？环境/数据/红线约束（Cowork 沙箱 / 数据真实性铁律 / G-X 硬坑）？
- **算清**：MVP 边界清晰 + 选定一个方向（大方向由 Doctor 拍，CC 给选项与 trade-off）+ 列明约束。
- **打回信号**：范围无边界；"顺便把 Y 也做了"（膨胀）；方案与问题不匹配；无视已知环境硬约束。

### 维度三 · 交付与验收（怎么算完、怎么验、不做什么）
- **要问**：怎么算做完？哪几条是**硬验收**？每条怎么验（命令 / 产物 / Doctor 测）？明确**不做**什么？谁验收、何时验？
- **算清**：每条验收都有客观口径（可机器验证或明确由 Doctor 测）+ 划定非交付项。
- **打回信号**：验收主观（"做好了就行"）；没有可验证口径；没划非交付项（范围必漂）。

## §6 · 与既有机制联动
- **G-X4 / prd-acceptance-template**：本 skill 是 G-X4 的"对齐器 + 起草器"，治理铁律全继承模板 §六/§七/§八。
- **AskUserQuestion**：Step 2b 选择题对齐的主力工具。
- **brain-save**：执行结束 /save 贴 PRD 路径 + 状态（brain-save Step 6）。
- **TaskList**：PRD 管"什么算 done"，TaskList 管"怎么做"，并行不冲突。
- **未来可拆**：若对齐问题库长到独立成势，可分出 `brain-spec`（需求对齐）+ brain-prd（起草）两 skill；当前合一够用。

## §7 · 边界
- 模板缺失 → 用 G-X4 §四结构 fallback，仍守三态 + 不打 ✓。
- 任务很清晰（Doctor 已把需求/验收说全）→ 对齐可压缩为"一次复述确认"，不强行多轮。
- 任务中途缩成小改 → 可撤 PRD（只 Doctor 决定）。
- checkpoints/ 不存在 → 先建。
