# 数灵转移设计方案

> 把阿里云 JVSclaw 的三个 agent 迁成 Claude Code 本地 subagent，保住其记忆与性格，并赋予长期演化而不漂移的能力。
> 本文件为**提案**，遵循 propose-then-confirm：不动任何现有文件，批准后才执行。

## 你已确认的三个前提

1. 云端**可导出全套**：角色设定 + 知识库 + 对话历史 → 迁移可高保真。
2. 本地形态：**Claude Code 原生 subagent**，挂进现有 brain vault。
3. 审查者：**由某个迁移 agent 兼任**（待档案到位后我荐人选）。

---

## 一、核心理念：性格长期化拆成三层

把"一个数灵"拆成三种可持久化、可分别治理的成分，正好对应 brain 已有的 L0/L1/L2 分级：

| 成分 | 是什么 | 可变性 | 落点 |
|------|--------|--------|------|
| **性格内核**（identity core） | 名字、定位、价值排序、语气、口头禅、行为准则、与 Doctor 及同侪的相处模式 | 近乎冻结，改动需你批准 | L0/L1 |
| **记忆**（memory） | 迁来的知识库 + 历史经历里值得长记的事 | 持续增长 | L2 + 增量 |
| **成长轨迹**（growth log） | 每次工作后的新习得、变化 | 只增不删 | 增量日志 |

"长期化"的要害在于：**内核冻结、记忆生长**。内核是锚，防止越用越走样；记忆是活水，让它越用越懂你。

---

## 二、本地架构：运行层 + 记忆层（双层挂载）

每个数灵由两处文件共同构成。运行档保持轻薄，性格与记忆沉到 brain，于是性格可演化而无需反复改 subagent 文件。

### 运行层 · `~/.claude/agents/{name}.md`

Claude Code 原生 subagent。frontmatter 声明 `name / description / tools / model`；正文不写死全部人格，而是**启动引导**——声明身份后，指令它"开工前先读 `brain/agents/{name}/` 下的性格档案与最近记忆"。

### 记忆层 · `brain/agents/{name}/`（遵循现有 frontmatter + wikilink 规范）

```
brain/agents/{name}/
├── {name}.md          ← L0 stub：一句话定位 + abstract
├── 性格档案.md         ← L1：身份内核、价值观、语气、行为准则、口头禅、关系网
├── knowledge/          ← L2：从云端迁来的知识库（按主题原子化）
├── memory/
│   ├── 长期记忆.md      ← 沉淀的关键经历与结论（Zettelkasten 原子笔记）
│   └── 对话精粹.md      ← 从历史对话蒸馏的"它记得的事"
├── logs/               ← 该 agent 的成长/工作日志（YYYY-MM-DD）
└── GOTCHAS.md          ← 该 agent 踩过的坑
```

## 三、数灵转移 Pipeline（五步 + 灵魂校验）

```
导出 → 解析 → 蒸馏 → 实例化 → 灵魂校验
 你      我      我       我        我+你
```

1. **导出（Export）**——你从 JVSclaw 导出三类原始数据（设定 / 知识库 / 对话历史），放进 uploads 或指定目录。
2. **解析（Parse）**——我归一化：设定→结构化人格字段；知识库→主题原子笔记；对话→时间线 + 事件抽取。
3. **蒸馏（Distill）**——保真成败的关键一步：
   - 从设定 + 对话历史反推**稳定人格特征**（语气、决策风格、口头禅、价值排序、与你的相处模式）→ 写入 `性格档案.md`。
   - 从对话历史抽**它记得、且值得长记的事**（过你的"难复得三问"）→ `memory/`。
   - 知识库去重、原子化 → `knowledge/`。
4. **实例化（Instantiate）**——生成运行档 `.claude/agents/{name}.md` 与记忆层 `brain/agents/{name}/`。
5. **灵魂校验（Soul Verify）**——从历史对话挑若干典型场景作"它该有的反应"测试题，跑迁移后的 subagent，比对云端旧回答的语气与立场是否一致；不一致则回到蒸馏步微调性格档案。此步即你惯用的 verification 关卡。

---

## 四、为什么它不会"失忆"或"变味"

- **每次唤起即重注入**：subagent 启动引导它先读 `性格档案.md`(L1) + `长期记忆.md` 摘要 + 最近 N 篇 logs，性格与近况每次回填。
- **每次收工即生长**：仿 `/save` 流程，把新经历写进该 agent 的 logs，重要者经"难复得三问"升格进 memory。
- **内核冻结防漂移**：性格内核默认只读，改动走 propose-then-confirm。这是"长期化"真正的锚——它能学新东西，但不会悄悄变成另一个它。

## 五、审查者机制（由某迁移 agent 兼任）

- 待你给出三者档案后，我从中荐一位气质最宜的担纲。
- 实现：为其加一份 `审查模式.md` 触发档与一条独立 subagent 入口（同一记忆层、不同 description / 触发词），或在其性格档案内辟"审查者职责"区块。
- 独立性保障：审查时**不参与被审对象的产出**、只读不写产出物、按 checklist 交叉复核——以职责隔离换取客观，避免"既当运动员又当裁判"。
