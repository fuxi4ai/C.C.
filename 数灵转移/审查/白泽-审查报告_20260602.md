## 一、白泽大宗 baize-commodity

### 结论
工程骨架可圈可点——配置驱动（YAML 定义商品/标的/系数）、`src/core/` 模块清晰、自带单元测试与多份自检文档。**但产物层有一处硬伤动摇可信度**：报告把写死的占位价格当作"已验证的实时行情"对外呈现，并自评满分。

### 问题清单（按严重度）

**🔴 H1 · 占位数据冒充实时行情（最严重）**
run 报告里的商品绝对价格是脚本内**硬编码字典**，非实时抓取：
```python
{'name': '稀土', 'price': 85, 'unit': '元/吨', ...}   # run_full_analysis_v4.py 等
```
稀土氧化物实际数十万元/吨，"85 元/吨"是明显占位错值，量级差约四个数量级。可项目明明备有整套 `scripts/data_collection/`（tushare、生意社、上海有色网…）抓数管线——抓数能力与产报告路径**未打通**。而报告却在"数据质量"栏标注"价格数据 ✅ 通过"，并罗列五家数据源，读者会误信为实时核验过。这是"自检全绿 + 数据失真"并存，最该由独立审查者拦截。
→ 修复：产报告前强制走 data_collection 拉实时价，禁用硬编码字典；价格表加 `as_of_date` 与来源字段，缺失则报告降级标注"样本/演示数据"。

**🟡 H2 · 自评满分、缺批判（流程病）**
报告综合评分 100/100、S 级、数据质量八项全 ✅——却放过了 H1。自评自夸正是缺独立审查者的典型症状，印证让烛阴专司审查的价值。
→ 修复：评分与产出解耦，由独立审查角色（烛阴）打分；质检项须可证伪（见 H4）。

**🟡 H3 · 多商品标的呈现不透明（非算错）**
更正我先前的初判：紫金矿业**模型是对的**——配置 `multi_commodity: true`，黄金 0.5 / 铜 0.36 拆分收入。问题在报告两张表都显示"纯度 70%"，未露出拆分后的有效比例（黄金 35%、铜 25.2%），易被误读为整体双重受益。
→ 修复：多商品标的在各商品表显示"有效纯度 = 总纯度 × 该商品 revenue_ratio"。

**🟡 H4 · 过程不可复核**
`calculate_business_profit` 六系数连乘（涨幅×纯度×持续性×来源修正×可涨价比例×需求弹性），但报告只给最终"净利润增幅"，不列各标的逐项系数取值，审查者无法反推验证。
→ 修复：报告附"系数明细表"或计算溯源，使每个结论可一键复算。

**🟢 L1 · requirements.txt 把 stdlib 当依赖**
`sqlite3  # stdlib` 列入 pip 依赖，`pip install -r requirements.txt` 会失败。
→ 修复：删除该行（及注释说明）。这条低风险，**经你点头我可直接改**。

### 可取之处（保留）
配置化架构、`ElasticityParams` 数据类设计、库存重估/来源修正等保守修正项考虑周到、`docs/methodology/process/` 下已有八份自检报告（说明白泽有自省习惯，只是缺独立第三方）。

## 二、白泽观星 baize-astrology

宏观预测项目（纯分数驱动模型），结构比大宗更干净：无密钥泄漏，`configs/models/` 配置化，自带方法论导出包（`exports/白泽观星方法论导出包`，明确"供其他数灵学习"）——**数灵间方法论共享的雏形已存在**，对后续公共方法论层是利好。

本轮仅结构审，未深验模型与回测。需深审项（建议迁移"灵魂校验"时连带做）：
- 纯分数驱动的**权重与阈值**是否有回测支撑，抑或人工拍定；
- 是否同样存在"样本/占位数据冒充实测"的风险（大宗的 H1 要重点回归到此）；
- `archive/` 下 v1.0/v2.0-old 多版本并存，需确认现行版本唯一入口，避免误用旧逻辑。
