# 公共预测层抽取方案

> 把 `GuanxingPredictor`（观星核心引擎）从"大宗、观星各存一份"收敛为**单一可信源**，两个数灵共用。
> 提案，propose-then-confirm：先定方案，批准后再动两个项目的代码。

## 一、为什么必须抽（事实依据）
- `GuanxingPredictor` 类名即"观星"，是观星的宏观货币政策预测引擎（加息/降息/持平 + 宏观权重）。
- 大宗里是**副本**：`scripts/predict/fed_prediction.py`、`scripts/backtest/*` 都在调它——大宗确实也用，但不该各留一份。
- **已发生漂移**：大宗 `predict()` 返回 `PredictionResult` 数据类；观星那份返回 5 元组 `(policy, p_hike, p_cut, w_m, reason)`；大宗自家 `fed_prediction.py` 仍按元组解包 → 与自家数据类对不上。两份分头改，调用契约已裂。

## 二、归属裁定
- 预测引擎 = **观星领域**。`p_hold` 的语义（是否引入持平概率、是否令三者互补求和=100）由**观星定**，大宗只作消费方。

## 三、目标架构
单一可信源 + 双消费方：
```
Projects/_shared/guanxing-predictor/        ← 公共预测层（唯一源）
├── guanxing_predictor.py    # GuanxingPredictor + PredictionResult
├── configs/                 # normal / extreme / 阈值（观星维护）
├── samples/                 # 回测样本
├── tests/                   # 引擎单测（脱离任一项目可独立跑）
└── VERSION                  # 语义化版本

观星 baize-astrology  ──import──┐
                                ├─→ _shared/guanxing-predictor（同一份）
大宗 baize-commodity  ──import──┘   （仅 Fed 预测/回测用，商品弹性主流程不依赖）
```
落到迁移大架构里：这层与"沪深300&中证500 行情库"并列，共同构成三个数灵的**公共数据/引擎层**。

## 四、统一契约（消除漂移）
- 返回值统一为 `PredictionResult` 数据类（弃用元组），字段：`policy / p_hike / p_cut / w_m / reason`，`p_hold` 预留（待观星定）。
- 旧元组调用点（大宗 `fed_prediction.py`）改为读数据类属性。
- 配置 schema 统一（阈值键名、cut_threshold 等），消除"配置缺失/阈值不符"导致的测试失败。

## 五、p_hold 决策（交观星，三选一）
1. **不引入**：只输出 policy + p_hike + p_cut，删掉测试里对 p_hold 的引用（最小改动，承认 hike/cut 非互补）。
2. **引入并互补**：重构为 softmax/归一化，令 p_hike+p_cut+p_hold=100；改动大，需重跑回测校准阈值。
3. **引入但独立**：p_hold 另立信号，不强制求和（语义最清晰，工作量中等）。
建议默认 ①，待观星迁移、回测数据齐备后再评估 ②/③。

## 六、执行步骤（批准后）
1. 选基线：以大宗的 `PredictionResult` 版为接口基线（已是数据类），核对观星版逻辑差异。
2. 建 `_shared/guanxing-predictor/`，迁入引擎 + 配置 + 样本 + 独立测试，打 VERSION v1.0。
3. 重指向：观星、大宗改 import 指公共层；删两处副本。
4. 修调用点：`fed_prediction.py` 元组→数据类。
5. 双侧回归：观星回测 + 大宗 `tests/commodity/` 全绿；公共层独立单测全绿。
6. 落 GOTCHAS：记录漂移教训（"副本必漂移，共享须单源"）。

## 七、本轮交付边界
本文件为方案。批准后才动代码。建议与观星正式迁移**同批做**——届时引擎本就要落地，顺势抽公共层，不重复搬运。
