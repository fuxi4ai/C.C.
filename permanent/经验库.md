### [EXP-20260529-001] P1-A/B/C/D 信源混合实证案例（中际旭创 CPO 转型）
**类别**: cases
**场景**: 评分/研究目标公司处于产业链快速演化期（如 CPO 替代可插拔光模块的转型期），各方信源 P 级混合——既有 P1-A 公司公告硬实证、又有 P1-C 财经媒体推算、还有 P1-D 单源风险。如何在评分/入库中合理处理。
**做法**: ① **P1-A 一手源**（公司公告/IPO 招股书/SEC/官方 datasheet）= 直接采信，作为评分维度核心证据；② **P1-B 权威机构**（帕米尔/Goldman/Yole）= 采信但允许在 _meta 加注"非公司官方"；③ **P1-C 多源转述**（一线财经媒体多源交叉）= 采信但 properties 标"行业估算"；④ **P1-D 单源**（行业站/单一财经媒体）= 仅作概念存在依据，**具体数字必须 P1-A/B 二次核实后入图谱**——若数字进入 evidence 必须显式标 `_meta.数据待P1B核实`。**实战示例**：中际旭创"1.6T CPO 月产 20 万只"P1-C 多源转述可入，但"通过 NVIDIA GB200 认证"P1-D 单源不应入硬关系边——按"已进入 CPO 量产爬坡阶段（具体认证状态需 P1-A 一手源核实）"措辞处理。
**为什么有效**: 区分"事实存在"（多源都说有 = P1-D 也可信）与"事实细节"（具体数字 = 必须 P1-A/B）两层；避免 P1-D 单源被无意识升格为 evidence；保留产业链快速演化期的研究敏捷性同时维护图谱事实严肃性。

### [EXP-20260624-008] 「P1 把 P2/P3 砍下来」：好看数字必回溯一手，异值降甄别注
**类别**: cases
**场景**: 核实国产替代/某公司产能·良率·订单等"好看数字"时，P2/P3（财经自媒体/媒体转述）给出的数与公司一手口径（董秘问答/年报/公告）差一个数量级或方向。
**做法**: 一律以一手为准、回溯坐实；记录取一手最新值，把 P2/P3 异值降为**甄别注**（不并为竞争事实）；未见一手坐实的"亮眼指标"宁可不录。实例：南大光电 ArF 现有产能——董秘 2026-01 口径 **50 吨/年**（P1），财经自媒体传"宁波 500 吨达产"（P3 未坐实），差 10×；良率 99.7%/缺陷 0.03 同属 P3 未坐实，本轮不录。
**为什么有效**: 防股价炒作期的夸大数字混入图谱事实；复用英维克/盛合"叙事不入边、只留甄别注"范式。
**来源**: → [[渊图]] logs/2026-06-24-渊图-光刻胶核实与ArF过关入库 · 同族 ERR-20260602-001 / FIX-20260619-001

### [EXP-20260626-002] 外部理论/思想家入框架的处置范式：盟友/对手定位 → 抽零件 → 隔离线 → guardrail
**类别**: patterns
**场景**: 把某个名家概念/外部理论引入一个有自身认识论地基的项目（PEC 尤甚，但渊图/龙鱼五力引入分析框架同理）。默认陷阱＝"名气大=自动丰富框架"。
**做法**: ①**地基层定位**先问"它在地基层是盟友还是对手"（如海德格尔 aletheia 真理论 vs PEC 康德-波普尔主客架构＝对手）——别假设深刻=有用；②**只抽可操作零件**，不整体照搬（海德格尔只抽前结构三分+效果历史意识+自反台账）；③**画隔离线**：明写什么挡在门外（真理论/基础存在论留对象层外）；④**配 guardrail**拦层级僭越（把分析者层概念塞进对象层当真理标准＝挡箭牌，挂 G-27 子条款）。
**为什么有效**: 把"引入"从崇拜/排斥的二元变成"按零件拆装+边界管理"，既不浪费好零件、也不被整套异质本体论污染地基。
**来源**: → [[PEC]] C-20 候选 + G-27 子条款 · logs/2026-06-26-PEC-自我澄明条款立案.md · 与 raw 三层规程（L1 作者原命题不可改）+ C-04/C-05 邻接（本条管"框架"入、那些管"语料/作者命题"入）

### [EXP-20260626-001] 设计系统的「页面原型层」：主题之下、按页面形态再固化一层语法
**类别**: patterns
**场景**: 已有一套主题级 design system（色/字/语义/阴影 tokens），要为某种**具体页面形态**（卡片页 / 列表页 / 看板）沉淀可复用模版时，怎么归位、不与主题打架、不重复造 tokens。
**做法**: 别新立一套主题。立在主题**之下**的「页面原型层」(page archetype)——复用主题 base tokens，只补该页面形态独有的语法：组件 + 布局 + 状态编码 + 动效 + 原则反模式。落 `design-system/` 与主题 styleguide **并列**，命名 `{主题}-{形态}-styleguide.html` + 配套去数据 `*-boilerplate.html`；README 显式标注「主题层 / 页面原型层」两级，原型层注明「base tokens 承 {主题}」。
**为什么有效**: 把"主题=视觉基因"与"原型=页面形态语法"两层解耦——每出一种新页面不必复制整套 tokens、也不污染主题；同一主题下可长出多种原型（暖色卡片页 / 暗色看板…），各自 styleguide 自洽。
**来源**: → [[O MY HTML]] `design-system/warm-cardpage-styleguide.html`（首个原型层·暖色卡片页）· logs/2026-06-26-暖色卡片页范式固化

### [EXP-20260622-001] Cowork 沙箱挂载盘 sqlite 写范式：/tmp 暂存回写 + journal 截断；hot journal→只读 readonly 诊断口诀
**类别**: patterns
**场景**: Cowork 沙箱内对挂载盘（~/Documents 映射）上的 .db 写入/ingest；或排查"库 integrity ok 却连 mode=ro 都打不开、报 readonly"。覆盖 brain 所有落在挂载盘的库（白泽 business_breakdown / 烛照 recap / Market-Data / DVA）。
**做法**: ① **写范式**：挂载盘"可建·可改·可截断·**禁删**"——sqlite 事务要建并删 rollback `-journal`，删不掉→`disk I/O error`。故别直写；直写撞 I/O/readonly 时自动「拷库到 /tmp →在可写副本上 ingest（顺带回滚旧 journal）→把挂载盘残留 `-journal` **截断为 0 字节**（禁删但可截断，0 字节=无需回滚·无害）→ /tmp 整库 `cp`/`copyfile` 覆盖回写（覆盖不涉删，挂载盘允许）」。真机跑则 journal 正常删除、更干净。② **诊断口诀**：库 integrity ok 却 `mode=ro` 报 `attempt to write a readonly database` → **先怀疑残留 hot `-journal`**（中止写留下），不是权限——sqlite 开库必先回放 journal 回滚、回滚要写、只读挡住即报此。应急：真机 `sqlite3 库 "PRAGMA integrity_check;"`（rw 触发回滚+删 journal）。③ 同款"禁删"也让沙箱 git 留 `.git/index.lock` 陈旧空锁 → 真机 `rm -f` 解。
**为什么有效**: 把"禁删但可覆盖/截断"这条挂载盘真约束翻成正向序列（覆盖代删、截断代删），绕开 sqlite 唯一需要删的环节；诊断口诀把"readonly"误导回指真因（hot journal 而非权限），省去每库重推一遍。zhuzhao 已独立用此范式，build_warehouse 2026-06-22 据此根治。
**来源**: → [[白泽大宗]] · GOTCHA-027/028 · build_warehouse.py /tmp 回写 · zhuzhao-market-fetch SKILL · logs/2026-06-22-看板星图金融线与白泽库断链根治

### [EXP-20260617-001] 真诚/履约鉴别：剔除互利无成本项 + costly signal 按可收回性加权
**类别**: patterns
**场景**: 判一方（国家/企业/人）的协定/让步/承诺是"真意"还是"假意/拖延"——任何"行为是否真诚"的显示偏好分析。
**做法**: 两步过滤再下判。① **剔除零鉴别力项**：对**双方有利、对发出方无成本**的条款（如美伊 MOU 的油气放行/解封锁——美得降油价、伊得收入），真意方与假意方**都会照做** → 对"真假"无鉴别力（信道退化），**不能当证据**。② **判定点只落在发出方须付代价的条款，且按"可收回性"加权**：可收回/cheap talk（口头背书、可逆豁免、可再加征）＝假意方照付、**弱鉴别**；**不可收回**（对方在己方履约前永久落袋、拆掉需数年重建的结构、真烧顶住反对的政治资本）＝假意方最不肯付、**强鉴别**。决定性证据＝**付下"若日后反悔则收不回"的代价**。
**为什么有效**: costly-signal 理论的精确版——信号的信息量 ∝ 成本 **且** 该成本对隐藏类型**不可退还**。假意型要省的不是"花没花成本"，而是"花下去收不回"的那部分；故唯有不可收回的代价能把真假分开。配套防坑：别把第三方出资（如 $300B 海湾私钱）当成发出方的代价——那是下游代理变量。
**来源**: → [[PEC]] · predictions-register IR-P7 · logs/2026-06-17-美伊MOU政评命题IR-P7立案与G-08适用边界

### [EXP-20260614-001] 案件合并/拆分判据：先看分析单元层次（主张层≠兑现层），层次不同则拆+互链
**类别**: patterns
**场景**: 两个 PEC 案（或任意两份资产）主题相邻、机制疑似同源，要判"合并成一案"还是"分立"。
**做法**: 先问**分析单元在哪一层**——是静态主张/法理基底（claim 层），还是动态行为/兑现执行（execution·显示偏好层）？再看**范围**（单海域 vs 跨海域、单文明 vs 跨文明）。① 层次或范围不同 → **不合并**：分立两案，用一条**共享机制候选**互链（feeder ↔ generalization），各供一个反向数据点。② 同层同范围、仅视角不同 → 可合并为一案 + 子节。合并前过 G-11（层次混淆）+ G-12（维度膨胀）两闸。
**为什么有效**: 合并把"主张"与"兑现"塞进同一容器 = G-11 层次混淆 + 容器范围错配；分立+互链既守层次、又不丢共享机制，符合 GE 目录"慎防维度膨胀"。

### [EXP-20260613-001] 大项目审计：多路只读子代理分维度并行 + 主代理汇总分级
**类别**: patterns
**场景**: 要审一个体量大的项目（几十文件、单文件几十 KB、GOTCHAS 几十 KB），一次性读不完也读不准。
**做法**: 按维度切 N 路（如 ①代码质量与健壮性 ②文档/架构一致性 ③已知坑回归），每路派一个**只读** general-purpose 子代理，prompt 里给清线索 + 要求"真实文件:行号、标严重度、不许编造"；主代理只汇总，合并成 🔴/🟡/🟢 分级报告落盘。子代理并发发出（一条消息多个 Agent 调用）。
**为什么有效**: 分维度让每个子代理上下文聚焦、互不挤占；只读避免误改；主代理拿结论不拿文件 dump，省上下文又得全局视角。

### [EXP-20260613-002] "假成功"反模式：async 不 await + 子进程 stdio:inherit
**类别**: patterns
**场景**: 排查"批量任务报告全成功，但实际没产出/没数据"这类诡异现象。
**做法**: 盯两个特征点——① 异步流水线被 fire-and-forget（漏 `await`），异常逃出 try/catch、调用方拿不到失败；② 该流程被 `spawnSync('node', …, {stdio:'inherit'})` 当子进程跑时，子进程会在异步活儿真正干完前就 `exit 0`，父进程据退出码记"ok"=假成功。修复=补 `await`+try/catch；验证=故意让子步骤抛错，看批量汇总是否如实标 fail。
**为什么有效**: 退出码只反映同步栈结束，不等于异步任务完成；把这两点连起来看，一眼定位"统计骗人"的根因。
**来源**: → [[DVA]] · dva.js H1 修复 · logs/2026-06-13-DVA审核修复与建仓

### [EXP-20260523-002] 修 error 会 unmask 同文件 warning
**类别**: patterns
**场景**: 用校验器（如 validate-frontmatter）批量修文件，修完一类问题后告警数反而"上涨"。
**做法**: 校验器常对同一文件既报 error（缺字段）又报 warn（值不合规），汇总按"有 error 即进 error 桶"。补完字段清掉 error 后，原被盖住的 warn 浮现、warn 数上升属正常。处理顺序：先拿 0 error 基线，再单独过 warn；别被"warn 变多"误判成"越修越糟"。
**为什么有效**: 区分"结构缺失(error)"与"值不合规(warn)"两级，warn 上浮是 unmask 而非新增。
**来源**: → logs/2026-05-23-brain校验清零与PEC中东油落盘

### [EXP-20260524-001] "revealed-preference 判别器"——把"X 是否吓退/奏效 Y"从情绪命题转成可观测判别器

### [EXP-20260531-001] 「框架前反思三步」——遇二元/谁赢谁输命题，先重设问题再分析
**类别**: patterns
**场景**: 收到「X vs Y」「谁赢谁输」「是不是该 A」类命题（含用户/输入语料自己内置的二元默认值与引导方向）时，易顺着被预设的框架跑、把分析力耗在二元内部选边。

### [EXP-20260619-002-P] 效率尺 ≠ 分配尺——同一批政策换尺会重排，别假设两序同向
**类别**: patterns

### [EXP-20260624-001-T] FUSE `.fuse_hidden` 孤儿分流：验头/验内容再处置，别凭直觉删
**类别**: tools
**场景**: Cowork 沙箱 FUSE 挂载盘里反复冒出 `.fuse_hidden*` 文件（WAL 库未干净关闭、文件被覆盖写时旧 inode 被改名遗留）。要清理时——它们**同名不同质**，混为一谈会误删活资产。

**场景**: 用 `@anthropic-ai/sdk` 接任何 Anthropic 协议兼容的模型（DeepSeek V4 Pro、各类推理模型），靠环境变量切 baseURL/model。

**为什么有效**: thinking/text 分块是推理模型的协议特征；按 type 取块对普通模型也兼容（只有 text 块）。一次写对，换模型不返工。

### [EXP-20260529-003-S] 二分法 → N 维场景化的方法论迁移逻辑
**类别**: strategies
**场景**: 产业链快速演化时，原本可用的"是 vs 否"二分法（如"自研芯片 vs 外购"、"模块组装 vs 光源芯片"、"高端 vs 低端"）开始系统性偏离产业链直觉——评分排名与产业链事实矛盾，专家或对方反驳"不公允"。
**做法**: ① **识别二分法失效信号**——评分排名与产业链直觉矛盾、对方反驳"不公允"、同一公司被不同视角评出截然不同结果；② **拆解二分法背后的隐含维度**——通常二分法把多个维度压缩到了一个"主轴"上。比如"模块组装方 vs 光源芯片方"实际上隐含了 8 个独立维度（自研能力 / 自给能力 / 良率 / 性能 / 量产 / 客户结构 / 工艺壁垒 / 叙事对冲）；③ **设计 N 维框架**（通常 6-10 维）：每维度独立打分 + 维度内 scoring_scaffold + 与主方法论归一映射；④ **验证维度正交性**——若多个维度高度相关，应合并或重设计；⑤ **首次实例化输出测试**——若 N 维结果与产业链直觉不矛盾且能解释二分法的失效原因，证明 N 维框架立住。
**为什么有效**: 二分法的本质是"信息压缩损失"——把高维产业结构压到单一主轴上必然丢失细节。N 维场景化是"按产业链特有维度展开评分"，让评分体系跟随产业链实际复杂度演化。配合主方法论 v1.1.1+ 子方法论范式样板。
**经典案例**: 2026-05-29 中际旭创/源杰科技 "模块组装 vs 光源芯片" 二分法被 8 维框架修正——中际旭创从 70 分升至 89（实际是垂直整合方而非被动模块方），源杰从 70.5 升至 75（光源稀缺被 80%+ 单一客户依赖部分对冲）。

### [EXP-20260529-004-S] 方法论进化路径的三类触发机制
**类别**: strategies
**场景**: 任何工作方法/评分体系/分析框架都需要进化。CC 应主动识别"何时该改、何时不该改"，避免被频繁干扰也避免僵化。
**做法**: 识别三类合法触发器（按 §0 进化引子元规则）：
- **触发 A · 外部成熟体系借鉴**：发现某项目/工具已经在同类问题上有成熟设计（如 v1.0 → v1.1 被龙鱼五力 v3.2.1 触发）→ 通常是**轻调**（v1.x → v1.x+1），借鉴 4-6 项可直接套用的设计
- **触发 B · Doctor/专家反驳**：当评分/分析结果与产业链直觉矛盾时，Doctor 反驳的不是个案而是方法论缺陷（如 v1.1 → v1.1.1 被"二分法不公允"触发）→ 通常是**子方法论扩展**（v1.x.1 试点），不重写主方法论但增加场景化扩展
- **触发 C · 产业链事件触发**：当产业链发生重大事件（如 EUV 解禁、CPO 量产、Hi-ONE 进入商业化）→ 通常是**重构**（v1.x → v2.0），需要新增维度或重设权重
**反进化边界**（同等重要）：
- 一两个个案与直觉背离 → 个案处理（_meta 注记），不改方法论
- Doctor 临时性偏好 → 临时调整本次输出，不入方法论
- 短期市场情绪 → 不影响结构性评分

## 单一真源模块根治「同一路径散多处」接缝债（2026-06-15）

**做法**: 当同一资源路径（如 DB 文件）被硬编码散落在十余个脚本、且写法各异（`ROOT/data/x`、`__file__.parents/data/x`、相对 `data/x`、dict 里相对）时——别逐处就地改（留 N 份拷贝＝下次再改又 N 处），而是建一个**单一真源模块**（如 `scripts/lib_paths.py:BUSINESS_DB`，带环境变量覆盖供测试），各脚本统一 `import` 它、保留原变量名（`DATABASE_PATH = BUSINESS_DB`）使下游零改动。统一引导：脚本均在 `scripts/<子目录>/<文件>.py` 时 `sys.path.insert(0, parents[1])` 恒指 `scripts/`，import 一致。
**为什么有效**: 一处改、处处生效；根治接缝债（[[通用教训]] GOTCHA-008）；附带修了相对路径的 cwd 依赖坑。代价：每脚本加几行 import 引导，需逐个测「独立跑不报错」（缺第三方依赖的用 stub 验证路径解析）。
**适用判据**: 路径/常量被 ≥3 处引用且会变 → 单一真源；只 1-2 处或永不变 → 不必过度抽象。

### [EXP-20260626-001-P] Cowork artifact「重渲即部署」：生成器末尾套 meta 头直写 index.html
**类别**: patterns
**场景**: 本地生成器（读本机 DB）产出 HTML 日报，要同步到 Cowork artifact；但生成器是 CLI、调不了 update_artifact 这个 Cowork 工具，又不想每次手工部署（手动＝易漏＝「坏了没人盯」）。
**做法**: 生成器末尾加部署步——把产出 HTML 在唯一 `<!DOCTYPE html>` 后注入 `<script type="application/json" id="cowork-artifact-meta">{name,schemaVersion,description}</script>`，直接 `write_text` 到 `~/Documents/Claude/Artifacts/{id}/index.html`。写前备份旧版、目录不存在则告警跳过、`--no-deploy`/`--output` 测试模式跳过。一次重渲同时更新 AI4ME 产物 + artifact。
**为什么有效**: Cowork artifact 渲染读的是 index.html 文件本身（已验证直写即生效），故本地直写＝部署，绕开「CLI 调不到 MCP 工具」的隔阂。代价：绕过 update_artifact，manifest 的 updatedAt 不刷新（内容仍更新），可接受。

### [EXP-20260626-002-P] 自制 skill「合并版」做精简主干 + references 渐进式披露，别拍平成胖文件

### [EXP-20260627-001-P] 图谱口径矛盾先判「粒度+时效错配」，别急着选 winner
**类别**: patterns
**场景**: 知识图谱里同一实体出现两套互斥口径（如盛合「不涉及950/960 CoWoS-L」vs「为海思供CoWoS-L含950PR」），第一反应想删一条/选权威的留。
**做法**: 先把两条记录的 `data_sources`（`data_vintage` + `confidence_level`）和**语义粒度**都挖出来再判。多数「矛盾」是两类错配而非真冲突：①**时效**——旧口径的 P1 源可能是一年前（本案旧值 P1=帕米尔2025-05-31，标 undated；新值 P1=2026-04~06），新覆旧；②**粒度**——两条说的其实不是一回事（950PR=入门级小尺寸 ≠ 950/960旗舰大尺寸，本就可共存）。判明后走**调和-精修**（把口径改成分粒度/带 as_of，保留全部边），而非删事实。
**为什么有效**: 「按边数/置信度自动选 survivor」会把对的边并进错实体（参渊图 ERR-20260602-001 张冠李戴铁律）；先核时效+粒度能把「假矛盾」无损拆开，符合可逆优先。
**判别尺**: 矛盾 → 先比 data_vintage（谁新）+ 拆语义粒度（是不是同一档）→ 真冲突才裁定，错配就精修。

### [EXP-20260627-002-P] 龙鱼五力批量评分入库的标准动作（playbook）
**类别**: patterns
**场景**: 给一组 A 股标的（如某产业链合作方）做龙鱼五力六维评分并存进个股库。
**标准动作**:
1. **引擎在 Doctor 终端跑**——`consumers/龙鱼五力/batch_score.py <ts_code...>`（沙箱连不上 Tushare·代理 403，历来「CC 备命令/Doctor 终端跑」）；ts_code 引擎自校验名称、防张冠李戴。产出 `batch_out/latest.json`（落在挂载盘，CC 可读）。
2. **CC 读 latest.json 打分**——`packages[].{stock,business,valuation_data,financial_score,kg_context,scoring_scaffold}`；**财务维(5)用引擎自动分**、其余五维 CC 按 RULES 打（技术供需/竞争**必须优先用 kg_context**，估值**用引擎实拉 PE/PB·禁凭记忆**），守兑现折扣 + 维度正交（负面只在一维扣）。
3. **诚实标 provenance**——CC 会话内分 `reviewer="research-CC"`、`source="engine实跑·CC会话内评分"`，**不冒充 DeepSeek 管线分**；库本就接纳混合 provenance（同一 record 多条 analyses 可不同 reviewer/date 共存）。
4. **入库 append 不覆盖**——`Database/龙鱼-标的分析库/records/{ts_code}_{name}.json`，已存在则 append 进 `analyses[]` + 更新 latest/updated_at；新建则全 schema（six_dim 键名固定带分值括号）。完事跑 `_index/build_index.py` 重建 INDEX；git 走 Doctor 终端（库有独立 .git）。
**为什么有效**: 把「沙箱取不到真财务」这条硬约束变成固定分工，既守数据真实性铁律（财务不编、非上市缺数留白），又让 CC 的定性判断（五维）有引擎+图谱双锚，provenance 可追、时序可叠。
