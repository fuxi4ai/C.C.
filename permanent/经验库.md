**来源**: → [[项目]] 或 logs/日期-主题

### [EXP-20260529-001] P1-A/B/C/D 信源混合实证案例（中际旭创 CPO 转型）
**类别**: cases
**场景**: 评分/研究目标公司处于产业链快速演化期（如 CPO 替代可插拔光模块的转型期），各方信源 P 级混合——既有 P1-A 公司公告硬实证、又有 P1-C 财经媒体推算、还有 P1-D 单源风险。如何在评分/入库中合理处理。
**做法**: ① **P1-A 一手源**（公司公告/IPO 招股书/SEC/官方 datasheet）= 直接采信，作为评分维度核心证据；② **P1-B 权威机构**（帕米尔/Goldman/Yole）= 采信但允许在 _meta 加注"非公司官方"；③ **P1-C 多源转述**（一线财经媒体多源交叉）= 采信但 properties 标"行业估算"；④ **P1-D 单源**（行业站/单一财经媒体）= 仅作概念存在依据，**具体数字必须 P1-A/B 二次核实后入图谱**——若数字进入 evidence 必须显式标 `_meta.数据待P1B核实`。**实战示例**：中际旭创"1.6T CPO 月产 20 万只"P1-C 多源转述可入，但"通过 NVIDIA GB200 认证"P1-D 单源不应入硬关系边——按"已进入 CPO 量产爬坡阶段（具体认证状态需 P1-A 一手源核实）"措辞处理。
**为什么有效**: 区分"事实存在"（多源都说有 = P1-D 也可信）与"事实细节"（具体数字 = 必须 P1-A/B）两层；避免 P1-D 单源被无意识升格为 evidence；保留产业链快速演化期的研究敏捷性同时维护图谱事实严肃性。
**来源**: → [[CPO时代企业评估框架]] §2 / [[渊图国产公司量化评分方法]] v1.1 §2.4 · logs/2026-05-29-渊图-CPO时代评估框架试点与v1.1.1落盘 · raw/韬定律中国公司量化评分-2026-05-28.md 附录 A

## patterns · 可复用流程 / 方法

> 跨项目都能套用的工作范式。

### [EXP-20260523-001] 分级加载（L0/L1/L2）控制上下文
**类别**: patterns
**场景**: 记忆/文档多到一次性读会污染上下文、又怕漏关键信息时。
**做法**: 每个单元拆三层——L0 一句话摘要（判相关性）、L1 概览（做"要不要深入"决策）、L2 全文（确认深入才读）。brain 里 = stub abstract / 系统概览 / architecture 深文件。
**为什么有效**: 大部分判断只需 L0/L1，省 token 又不丢可达性；OpenViking 实测输入 token 降 83–96%。
**来源**: → [[CLAUDE]] 「分级加载约定」· brain-anchors skill

### [EXP-20260523-002] 修 error 会 unmask 同文件 warning
**类别**: patterns
**场景**: 用校验器（如 validate-frontmatter）批量修文件，修完一类问题后告警数反而"上涨"。
**做法**: 校验器常对同一文件既报 error（缺字段）又报 warn（值不合规），汇总按"有 error 即进 error 桶"。补完字段清掉 error 后，原被盖住的 warn 浮现、warn 数上升属正常。处理顺序：先拿 0 error 基线，再单独过 warn；别被"warn 变多"误判成"越修越糟"。
**为什么有效**: 区分"结构缺失(error)"与"值不合规(warn)"两级，warn 上浮是 unmask 而非新增。
**来源**: → logs/2026-05-23-brain校验清零与PEC中东油落盘

### [EXP-20260524-001] "revealed-preference 判别器"——把"X 是否吓退/奏效 Y"从情绪命题转成可观测判别器
**类别**: patterns
**场景**: 遇到"威胁/惩罚/政策是否吓退/奏效"类命题（如"天价罚单是否吓退外资""制裁是否逼退对手"），易陷情绪化断言（"肯定吓退"/"根本没用"）。
**做法**: 不直接下断言，而是**找当事人用脚投票的可观测行为**当判别器。① 先拆前提混淆（如"罚单吓退 Tesla"——Tesla 实际是关税僵局离场、根本没被罚，两套机制别混）；② 找最干净的 revealed-preference 信号（苹果一边在德里高院打 $38B 反垄断官司、一边扩 iPhone 印度产能 → 罚单威胁没吓退其制造 FDI）；③ 把命题写成 dated 观察项（"苹果不论 CCI 结果到 2027 底是否继续扩产"），用未来行为裁定。
**为什么有效**: 行为(revealed preference)比表态/情绪更硬；判别器把不可证伪的情绪命题钉成可观测的 dated 预测，直接对抗 G-01（橡皮图章）+ G-08（"有威胁→必奏效"未论证跳跃）。
**来源**: → [[PEC]] CS-07 A09 / predictions-register IN-P3 · logs/2026-05-24-PEC-印度制造业立国窗口与营商环境

### [EXP-20260521-001-T] ASR 前先压缩音频
**类别**: tools
**场景**: 给 ASR（sensevoice-v1）准备音频、要走脆弱国内云链路上传时。
**做法**: 永远不传未压缩 WAV；先 ffmpeg 压成 AAC/m4a 16kHz 单声道（实测体积降到约 1/20）。异步任务（提交→轮询）每个网络调用都加重试。
**为什么有效**: 小文件在抖动链路上一次传完的概率大增；"缩小 payload"比"调网络"更治本。
**来源**: → [[DVA]] · INFRA-20260521-001

---

### [EXP-20260523-002-T] 重打包 Cowork .skill 包
**类别**: tools
**场景**: 改了 brain skill 的 SKILL.md，要把 `.skill` 安装包同步更新给 Doctor 重装。
**做法**: `.skill` = zip 包，布局为 `<name>/` + `<name>/SKILL.md`，压缩用 `ZIP_STORED`（对齐既有包）。沙盒挂载盘**禁 unlink**，覆盖既有包要"写临时文件 → `os.replace` 覆盖"（同目录 rename，允许）。打完用 `zipfile.testzip()` + 列 `namelist()` 验证。装包入口可用 present_files（.skill 自带 install 按钮）。
**为什么有效**: 匹配安装器预期布局；os.replace 绕开挂载盘 unlink 限制；先验证再交付不翻车。
**来源**: → logs/2026-05-23-记忆系统对标OpenViking与五项优化落地 · [[通用教训]]（沙盒只 rename 不 unlink）

### [EXP-20260523-003-T] 沙盒只读 git 姿势（不留 index.lock）
**类别**: tools
**场景**: 沙盒里要看 git 状态/历史，但 FUSE 挂载盘连只读 `git status` 都会留 `.git/index.lock` 且删不掉。
**做法**: 一律加 `--no-optional-locks`（如 `git --no-optional-locks log --oneline -5` / `status --porcelain`），它不刷新索引、不建锁。判断"自己刚改的文件是否已被他会话卷走"用 `git show HEAD:<path>` 比对磁盘内容；探测 .git 存在性优先用 Read/ls 而非 git。任何写操作贴命令给 Doctor（G-X2）。
**为什么有效**: `--no-optional-locks` 跳过"刷新索引"这一步——而正是刷新索引在尝试建 index.lock；`git show HEAD:` 直读对象库不碰工作区/索引。
**来源**: → [[通用教训]] G-X2补 / G-X9 · logs/2026-05-23-brain校验清零与PEC中东油落盘

## strategies · 打法 / 策略

> 解题策略、协作打法、元方法。

### [EXP-20260523-001-S] 对标外部方案先分"借/拒"，按自身规模与模型裁剪
**类别**: strategies
**场景**: 看到一个"更先进"的外部系统（如 OpenViking），想借鉴来改自己的系统。
**做法**: 先摸清自己系统的**真实**架构（读源文件不靠假设）→ 逐维度优劣对比 → 按**自身规模 + 协作模型**判断哪些值得借、哪些是过度工程，明确列出"不做"项及理由 → 分级提案待批。
**为什么有效**: 外部方案的设计前提（如自主 agent / 海量记忆）未必是你的前提；机械照搬会引入不匹配的复杂度。"明确不做"和"做"一样有价值。
**来源**: → [[OpenViking-Agent记忆系统]] · [[架构决策-20260514]] D6

### [EXP-20260523-002-S] 元数据批量修复"dry-run→批准→apply"范式
**类别**: strategies
**场景**: 要批量改一批文件的 frontmatter / 元数据（补字段、归一取值），既要快又怕改错或破坏既有内容。
**做法**: 一套逻辑写两遍——先写**只读 dry-run 脚本**逐文件算出拟改值并打印（旧→新），交 Doctor 过目；批准后把同一取值逻辑改成 **apply 脚本**落盘；最后**重跑校验器**验证清零。沙盒写失败的个别文件（FUSE 权限）用 Edit 工具兜底。取值规则透明化（日期取文件名/git、title 取 H1、tags/type 按路径推断）便于人工核。
**为什么有效**: dry-run 与 apply 共用同一函数 → Doctor 看到的就是将落盘的；只读预演把"批准"前置，把不可逆写操作关进人在环。
**来源**: → logs/2026-05-23-brain校验清零与PEC中东油落盘 · [[通用教训]]（G-X8 已落盘假象）

### [EXP-20260524-001-S] 闭环/万能命题遇反例 → 划"适用线域"，不推翻也不无视
**类别**: strategies
**场景**: 一个已沉淀的解释性闭环/强命题（如 CS-07 印度"现实失败→叙事胜利→继续投入"）遇到硬数据反例（iPhone 印度产能占全球约 1/4 = 真实物质捕获，不是"叙事胜利"）。
**做法**: 三选项里选第三条路——① 不强行把反例塞进闭环解释（那会让闭环能解释一切 = G-01 橡皮图章）；② 不因一个反例就推翻闭环（其原线域证据仍强）；③ **划定闭环的适用线域**：把"全域机制"降为"某线域机制"（闭环收窄为军事-外交线机制 1962/Sindoor，经济线存在真实捕获 = 部分证伪 + 命题收窄）。落盘时显式记"部分证伪 + 边界"，并 promote 回写源命题。
**为什么有效**: 既保住命题在有效线域的解释力、又诚实承认反例，是 G-10（框架自我保护）的**建设性防御**——框架靠"划边界"进化，而非靠"打补丁"续命（对接 predictions-register 更正协议的进步/退化判据）。
**来源**: → [[PEC]] 决策记录 2026-05-24 闭环分线边界裁定 · CS-07 A09 · logs/2026-05-24-PEC-印度制造业立国窗口与营商环境

### [EXP-20260528-001] 渊图信息甄别第三范式：技术能力叙事 vs 真实能力
**类别**: patterns
**场景**: 公司/产品被市场/媒体赋予某项技术能力叙事（如"华为完成 14nm 以上 EDA 全部国产替代"、"华为公开 7nm 级 EDA 工具链"），但 P1 一手源（官方稿/年报/招股书/产品发布会）从未确认。已有先例：英维克"信披诚信瑕疵"（叙事 vs 财报）、盛合晶微"叙事股权"（叙事 vs IPO 招股书）。
**做法**: 不入"虚假能力边"，改用 `_meta.信息甄别_<主题>_<date>` 标记：① `claim`（叙事原文）；② `verdict`（CC 判定：误传/部分真/无据）；③ `真实情况`（可考据的部分）；④ `evidence_sources`（P 分级，每条独立标）；⑤ `ingestion_decision`（不入边/仅 properties/仅 _meta）；⑥ `precedent`（指向同类历史范式）；⑦ `reviewer`。同时若叙事内核有真实部分（如华为有自研 EDA 努力但范围未达叙事），可在节点 description 留甄别注记 + 单独入"真实部分"边。
**为什么有效**: 渊图作为"慢更新长效产业知识库"，最大风险是把市场叙事当事实。三范式（英维克信披瑕疵 / 盛合晶微叙事股权 / 韬定律技术能力叙事）覆盖了财报-股权-技术能力三大叙事高发区；统一 formula 让后续遇同类直接复用，避免重复设计 schema。
**来源**: → [[渊图]] 决策记录 2026-05-23 英维克 / 2026-05-28 盛合晶微 / 2026-05-28 EDA 子图入库 · logs/2026-05-28-渊图-韬定律EDA子图研究汇报

### [EXP-20260529-002] 可插拔光模块 → CPO 产线兼容性 4 档复用度模型 + 资本开支转化效率公式
**类别**: patterns
**场景**: 产业链转型期评估某公司"老产能 → 新形态"的实际转化能力（不只是"是否做新业务"二分判断）。光通信 CPO 是首例，但模板可推广到先进封装（CoWoS-L 兼容性）/ GAA 工艺升级 / 国产替代任何产线改造场景。
**做法**: ① 按工艺段拆解列出公司各段（如光通信：硅光晶圆 fab / CW 激光器 / 调制器封装 / fiber attach / DSP 集成 / 测试与验证 / 组装自动化）；② 每段标注 4 档复用度——**完全可复用**（工艺路线本质相同，如自研硅光芯片可用于可插拔和 CPO 两形态）、**部分可复用**（需新增/升级设备，如对位精度需提升一个数量级）、**不可复用**（被替代环节，如可插拔 DSP 在 CPO 时代被 ASIC 吸收）、**新增**（CPO 时代特有需求，如 system-level validation）；③ 按段权重估算"复用度加权百分比"（中际旭创案例：30-40% 完全 + 30-40% 部分 + 20-30% 不可 + 20% 新增）；④ 用"单位产能投资额"做横向比较——中际旭创铜陵三期 4.52 亿 / 120 万只 = 377 元/只 capacity；按"完全复用 35% + 部分复用 35% + 不可复用 30%"权重推算 CPO 单线投资强度 ≈ 600-750 元/只（原产线 1.5-2 倍）。
**为什么有效**: 替代"是/否做新业务"的二分判断，把工程实证（产线/设备/工艺/资本）的传导转换为可量化分级。让产业转型评估摆脱"叙事性判断"进入"工程性判断"。配合主方法论 v1.1.1+ 子方法论范式使用。
**来源**: → [[CPO时代企业评估框架]] §1+§2 · logs/2026-05-29-渊图-CPO时代评估框架试点与v1.1.1落盘

### [EXP-20260528-002] 学术机构作为 originator 入库范式扩展（MIT/PoliBa → 北大）
**类别**: patterns
**场景**: 渊图主要收商业化供应链，但前沿技术的真正源头常在学术机构（如 MIT 电液纤维肌肉、北大 LogicFolding EDA）。直接用"university 类型节点"会污染 schema；不入又丢真实因果。
**做法**: ① 学术机构用 `company` 节点 + name 后缀注明性质（如 `company_MITMediaLab` / `company_PekingUniversityIC`）；② 与其产品的关系用 `enables` 边（不用 `supplies`——学术机构非商业供应商）；③ 性能指标入 node `properties` + 显式 `stage="实验室/研究阶段"`；④ 与已商业化路线的关系用 `competes_with` + 低 weight（0.55-0.6）体现成熟度差；⑤ "潜在应用"用 `used_in` + 低 weight + hedged 描述，不夸大为实际使用。
**为什么有效**: 学术 originator 在产业链里属"上游能力解锁"语义，enables 边正好承接；低 weight + stage properties 让前沿技术与成熟商业化路线在同一图谱里共存而不混淆；后续若该技术商业化（如出现 supplies 边的产业玩家），范式可向"研究→产业"过渡而无需 schema 变更。
**来源**: → [[渊图]] 决策记录 2026-05-23 MIT 电液纤维肌肉 / 2026-05-28 北大 LogicFolding EDA · logs/2026-05-28-渊图-韬定律EDA子图研究汇报

### [EXP-20260529-003-S] 二分法 → N 维场景化的方法论迁移逻辑
**类别**: strategies
**场景**: 产业链快速演化时，原本可用的"是 vs 否"二分法（如"自研芯片 vs 外购"、"模块组装 vs 光源芯片"、"高端 vs 低端"）开始系统性偏离产业链直觉——评分排名与产业链事实矛盾，专家或对方反驳"不公允"。
**做法**: ① **识别二分法失效信号**——评分排名与产业链直觉矛盾、对方反驳"不公允"、同一公司被不同视角评出截然不同结果；② **拆解二分法背后的隐含维度**——通常二分法把多个维度压缩到了一个"主轴"上。比如"模块组装方 vs 光源芯片方"实际上隐含了 8 个独立维度（自研能力 / 自给能力 / 良率 / 性能 / 量产 / 客户结构 / 工艺壁垒 / 叙事对冲）；③ **设计 N 维框架**（通常 6-10 维）：每维度独立打分 + 维度内 scoring_scaffold + 与主方法论归一映射；④ **验证维度正交性**——若多个维度高度相关，应合并或重设计；⑤ **首次实例化输出测试**——若 N 维结果与产业链直觉不矛盾且能解释二分法的失效原因，证明 N 维框架立住。
**为什么有效**: 二分法的本质是"信息压缩损失"——把高维产业结构压到单一主轴上必然丢失细节。N 维场景化是"按产业链特有维度展开评分"，让评分体系跟随产业链实际复杂度演化。配合主方法论 v1.1.1+ 子方法论范式样板。
**经典案例**: 2026-05-29 中际旭创/源杰科技 "模块组装 vs 光源芯片" 二分法被 8 维框架修正——中际旭创从 70 分升至 89（实际是垂直整合方而非被动模块方），源杰从 70.5 升至 75（光源稀缺被 80%+ 单一客户依赖部分对冲）。

### [EXP-20260529-004-S] 方法论进化路径的三类触发机制
**类别**: strategies
**场景**: 任何工作方法/评分体系/分析框架都需要进化。CC 应主动识别"何时该改、何时不该改"，避免被频繁干扰也避免僵化。
**做法**: 识别三类合法触发器（按 §0 进化引子元规则）：
- **触发 A · 外部成熟体系借鉴**：发现某项目/工具已经在同类问题上有成熟设计（如 v1.0 → v1.1 被龙鱼五力 v3.2.1 触发）→ 通常是**轻调**（v1.x → v1.x+1），借鉴 4-6 项可直接套用的设计
- **触发 B · Doctor/专家反驳**：当评分/分析结果与产业链直觉矛盾时，Doctor 反驳的不是个案而是方法论缺陷（如 v1.1 → v1.1.1 被"二分法不公允"触发）→ 通常是**子方法论扩展**（v1.x.1 试点），不重写主方法论但增加场景化扩展
- **触发 C · 产业链事件触发**：当产业链发生重大事件（如 EUV 解禁、CPO 量产、Hi-ONE 进入商业化）→ 通常是**重构**（v1.x → v2.0），需要新增维度或重设权重
**反进化边界**（同等重要）：
- 一两个个案与直觉背离 → 个案处理（_meta 注记），不改方法论
- Doctor 临时性偏好 → 临时调整本次输出，不入方法论
- 短期市场情绪 → 不影响结构性评分
**为什么有效**: 显式化"进化触发器"让方法论的迭代节奏可控——既不僵化（合法触发就进化）也不浮躁（不合法触发就个案处理）。三类触发机制覆盖了方法论演化的所有合理来源。
