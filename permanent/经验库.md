### [EXP-20260529-001] P1-A/B/C/D 信源混合实证案例（中际旭创 CPO 转型）
**类别**: cases
**场景**: 评分/研究目标公司处于产业链快速演化期（如 CPO 替代可插拔光模块的转型期），各方信源 P 级混合——既有 P1-A 公司公告硬实证、又有 P1-C 财经媒体推算、还有 P1-D 单源风险。如何在评分/入库中合理处理。
**做法**: ① **P1-A 一手源**（公司公告/IPO 招股书/SEC/官方 datasheet）= 直接采信，作为评分维度核心证据；② **P1-B 权威机构**（帕米尔/Goldman/Yole）= 采信但允许在 _meta 加注"非公司官方"；③ **P1-C 多源转述**（一线财经媒体多源交叉）= 采信但 properties 标"行业估算"；④ **P1-D 单源**（行业站/单一财经媒体）= 仅作概念存在依据，**具体数字必须 P1-A/B 二次核实后入图谱**——若数字进入 evidence 必须显式标 `_meta.数据待P1B核实`。**实战示例**：中际旭创"1.6T CPO 月产 20 万只"P1-C 多源转述可入，但"通过 NVIDIA GB200 认证"P1-D 单源不应入硬关系边——按"已进入 CPO 量产爬坡阶段（具体认证状态需 P1-A 一手源核实）"措辞处理。
**为什么有效**: 区分"事实存在"（多源都说有 = P1-D 也可信）与"事实细节"（具体数字 = 必须 P1-A/B）两层；避免 P1-D 单源被无意识升格为 evidence；保留产业链快速演化期的研究敏捷性同时维护图谱事实严肃性。

### [EXP-20260523-001] 分级加载（L0/L1/L2）控制上下文
**类别**: patterns
**场景**: 记忆/文档多到一次性读会污染上下文、又怕漏关键信息时。
**做法**: 每个单元拆三层——L0 一句话摘要（判相关性）、L1 概览（做"要不要深入"决策）、L2 全文（确认深入才读）。brain 里 = stub abstract / 系统概览 / architecture 深文件。
**为什么有效**: 大部分判断只需 L0/L1，省 token 又不丢可达性；OpenViking 实测输入 token 降 83–96%。

### [EXP-20260523-002] 修 error 会 unmask 同文件 warning
**类别**: patterns
**场景**: 用校验器（如 validate-frontmatter）批量修文件，修完一类问题后告警数反而"上涨"。
**做法**: 校验器常对同一文件既报 error（缺字段）又报 warn（值不合规），汇总按"有 error 即进 error 桶"。补完字段清掉 error 后，原被盖住的 warn 浮现、warn 数上升属正常。处理顺序：先拿 0 error 基线，再单独过 warn；别被"warn 变多"误判成"越修越糟"。
**为什么有效**: 区分"结构缺失(error)"与"值不合规(warn)"两级，warn 上浮是 unmask 而非新增。
**来源**: → logs/2026-05-23-brain校验清零与PEC中东油落盘

### [EXP-20260524-001] "revealed-preference 判别器"——把"X 是否吓退/奏效 Y"从情绪命题转成可观测判别器
**类别**: patterns
**场景**: 遇到"威胁/惩罚/政策是否吓退/奏效"类命题（如"天价罚单是否吓退外资""制裁是否逼退对手"），易陷情绪化断言（"肯定吓退"/"根本没用"）。
**做法**: 不直接下断言，而是**找当事人用脚投票的可观测行为**当判别器。① 先拆前提混淆（如"罚单吓退 Tesla"——Tesla 实际是关税僵局离场、根本没被罚，两套机制别混）；② 找最干净的 revealed-preference 信号（苹果一边在德里高院打 $38B 反垄断官司、一边扩 iPhone 印度产能 → 罚单威胁没吓退其制造 FDI）；③ 把命题写成 dated 观察项（"苹果不论 CCI 结果到 2027 底是否继续扩产"），用未来行为裁定。
**为什么有效**: 行为(revealed preference)比表态/情绪更硬；判别器把不可证伪的情绪命题钉成可观测的 dated 预测，直接对抗 G-01（橡皮图章）+ G-08（"有威胁→必奏效"未论证跳跃）。

### [EXP-20260531-001] 「框架前反思三步」——遇二元/谁赢谁输命题，先重设问题再分析
**类别**: patterns
**场景**: 收到「X vs Y」「谁赢谁输」「是不是该 A」类命题（含用户/输入语料自己内置的二元默认值与引导方向）时，易顺着被预设的框架跑、把分析力耗在二元内部选边。
**做法**: 进入分析**前**先写一行框架反思——① **共享前提＝**？（A、B 双方默认共享、未言明的那个命题）② **还成立吗？**（换时代/场景/行为体目标函数后是否已部分过时）③ **更值轴＝**？（有没有一条轴使 A vs B 降为该轴上的派生现象）。然后**承诺到更值的轴**，而非选一侧。守栏：二元有时正是对的框（如热战中"谁控咽喉点"确实决定性），"所有二元都是假问题"是另一个懒默认——要的是**先反思再决定**，不是反射性否定二元。
**为什么有效**: "不走平衡木"与"跳出二元"的合题＝最高价值的观点常是**重设问题**而非选边；防 frame capture（问题在分析开始前就被输入框架设错），与"概率上调"型偏差正交。活体：海权vs陆权专项——"海vs陆"共享"物理控线=权力"前提，跳出轴=斯特兰奇结构性权力（控规则 > 控路线）。
**来源**: → [[PEC]] GOTCHAS G-26（输入框架的二元默认陷阱·独立立案）· logs/2026-05-31-PEC-海权vs陆权专项与G-26立案

### [EXP-20260603-001-P] seed/import 环境分工：纯写文件本地做，依赖 os.homedir/外网的步骤交 Mac
**类别**: patterns
**场景**: 多环境（沙盒 CC vs Mac 终端）协作处理一条流水线，某些步骤能在沙盒做、某些必须在 Mac 做时，如何切分以又快又稳。

### [EXP-20260605-001] 子进程隔离批量编排带 process.exit 的旧命令
**类别**: patterns
**场景**: 要写一个「对一批对象循环跑某条已有命令」的批量编排器，但被复用的那条命令内部到处 `process.exit(1)`（错误即退进程）。在主进程里直接 `await handleX()` 会让单个对象的失败把整批进程一起带走。
**做法**: 不在主进程内联调用，改对每个对象 `spawnSync('node', [__filename, '子命令', ...args], {stdio:'inherit'})` 起独立子进程。子命令里的 `process.exit` 只杀子进程，主编排进程捕获 `status` 继续下一个，末尾汇总成功/失败。
**为什么有效**: 进程边界天然隔离 `process.exit`；零改动复用老命令（不必为批量去重构它的退出逻辑），又拿到逐对象容错 + 汇总。
**来源**: → [[DVA]] `dva update-all`（常更表批量更新）· logs/2026-06-05-DVA常更表与DeepSeek接入

### [EXP-20260606-001-P] 迁移残壳识别 + config 单一可信源 repoint 范式
**类别**: patterns
**场景**: 项目数据/资产被搬到新位置（云→本地、入公共数据层），但代码没跟着迁——成了"迁移残壳"。
**征兆三连**: ① 脚本写死旧机绝对路径（如 `/home/admin/...`）、且路径约定多套并存互相矛盾；② 相对导入（`from .x`）使脚本直接运行即 ImportError，整条链本就跑不起来；③ 文档数字与真库脱节（旧 README 称 128 条，真库 153）。残壳里 `.fuse_hidden` 残骸＝数据其实已搬走的旁证。
**修法五步**: 建根目录 `config.py` 路径单一可信源（向上探测定位新数据层 + `__main__` 自检全 ✅）→ 写脚本批量改**只改赋值 RHS、保留变量名**（`DB_PATH = config.RECAP_DB`），每处 `assert old.count==1` 防误伤 → 一次性/退役脚本归档 `_legacy/` + MANIFEST → 刷文档到真库实测数 → 逐模块 runtime import（不只 py_compile，编译不抓相对导入错）+ 跑通真库冒烟。
**为什么有效**: 路径收敛到一处，换机/再迁不改代码；runtime import 才暴露 py_compile 漏掉的导入级 bug；归档而非删除保留可逆。

### [EXP-20260606-002-P] 混沌非平稳系统的方法论立场：探测器>预测器、相对基线、特征>模型
**类别**: patterns
**场景**: 给股市这类混沌、复杂度过高、且"边缘会被市场适应淘汰"的非平稳系统建分析工具时，如何不沦为又一个很快失效的预测策略。
**做法**: 三条规格——① **做风险状态探测器，不做预测器**：输出"风险倾斜/前向回撤分布右移"，不输出点位或日期，把目标收到"能做的那件事"。② **一切相对滚动基线（分位/z），不用绝对阈值**：因为绝对+静态阈值正是被"市场适应"淘汰的东西（cycle_quant 全冰点的病根）。③ **复杂留给特征，简单留给规则，不上 ML**：非平稳小样本里过拟合是更大的敌人，且风险信号要敢照着动作必须可解释。验证用**条件分布右移 + walk-forward 样本外**，不用胜率（永远和真实基准率比，不和 50% 比）。
**为什么有效**: 把"无真理+边缘衰减"的认识论从泄气话翻译成可执行规格；适用于任何反身性/被套利的预测场景，不止股市。

### [EXP-20260606-004-P] 接管线先盘源能力、再下 gate，别上来就写
**类别**: patterns
**场景**: 要做"从 X 自动提取 Y"的管线（X=知识图谱/数据库/API）。

### [EXP-20260607-002-P] 「涨价提振却股价下行」三层背离归因法
**类别**: patterns
**场景**: 某标的在"利好/涨价提振逻辑"里，股价却持续下行，需解释背离。
**做法**: 拆三层归因——① **商品/行业层(β)**：利好是否已实现并 price in？是否见顶回调(仓单/库存/监管/需求阈值)？② **个股层(α)**：有无**量价背离**(价涨但自身产量/份额/出货因停产/换证/减持而落空，享价不享量)？③ **模型层**：用的是否静态弹性(利润敏感度)≠股价(预期贴现)？输入是否样本/滞后价？
**为什么有效**: 把"模型说该涨、市场却跌"的矛盾，定位到"已price in / 量价背离 / 模型口径"三个互斥来源，避免笼统归因；量价背离是最易被涨价叙事掩盖的 α 利空。

### [EXP-20260523-003-T] 沙盒只读 git 姿势（不留 index.lock）
**类别**: tools
**场景**: 沙盒里要看 git 状态/历史，但 FUSE 挂载盘连只读 `git status` 都会留 `.git/index.lock` 且删不掉。
**做法**: 一律加 `--no-optional-locks`（如 `git --no-optional-locks log --oneline -5` / `status --porcelain`），它不刷新索引、不建锁。判断"自己刚改的文件是否已被他会话卷走"用 `git show HEAD:<path>` 比对磁盘内容；探测 .git 存在性优先用 Read/ls 而非 git。任何写操作贴命令给 Doctor（G-X2）。

### [EXP-20260528-001] 渊图信息甄别第三范式：技术能力叙事 vs 真实能力
**类别**: patterns
**场景**: 公司/产品被市场/媒体赋予某项技术能力叙事（如"华为完成 14nm 以上 EDA 全部国产替代"、"华为公开 7nm 级 EDA 工具链"），但 P1 一手源（官方稿/年报/招股书/产品发布会）从未确认。已有先例：英维克"信披诚信瑕疵"（叙事 vs 财报）、盛合晶微"叙事股权"（叙事 vs IPO 招股书）。

### [EXP-20260529-003-S] 二分法 → N 维场景化的方法论迁移逻辑
**类别**: strategies
**场景**: 产业链快速演化时，原本可用的"是 vs 否"二分法（如"自研芯片 vs 外购"、"模块组装 vs 光源芯片"、"高端 vs 低端"）开始系统性偏离产业链直觉——评分排名与产业链事实矛盾，专家或对方反驳"不公允"。
**做法**: ① **识别二分法失效信号**——评分排名与产业链直觉矛盾、对方反驳"不公允"、同一公司被不同视角评出截然不同结果；② **拆解二分法背后的隐含维度**——通常二分法把多个维度压缩到了一个"主轴"上。比如"模块组装方 vs 光源芯片方"实际上隐含了 8 个独立维度（自研能力 / 自给能力 / 良率 / 性能 / 量产 / 客户结构 / 工艺壁垒 / 叙事对冲）；③ **设计 N 维框架**（通常 6-10 维）：每维度独立打分 + 维度内 scoring_scaffold + 与主方法论归一映射；④ **验证维度正交性**——若多个维度高度相关，应合并或重设计；⑤ **首次实例化输出测试**——若 N 维结果与产业链直觉不矛盾且能解释二分法的失效原因，证明 N 维框架立住。
**为什么有效**: 二分法的本质是"信息压缩损失"——把高维产业结构压到单一主轴上必然丢失细节。N 维场景化是"按产业链特有维度展开评分"，让评分体系跟随产业链实际复杂度演化。配合主方法论 v1.1.1+ 子方法论范式样板。
**经典案例**: 2026-05-29 中际旭创/源杰科技 "模块组装 vs 光源芯片" 二分法被 8 维框架修正——中际旭创从 70 分升至 89（实际是垂直整合方而非被动模块方），源杰从 70.5 升至 75（光源稀缺被 80%+ 单一客户依赖部分对冲）。

### [EXP-20260529-004-S] 方法论进化路径的三类触发机制
**类别**: strategies
**场景**: 任何工作方法/评分体系/分析框架都需要进化。CC 应主动识别"何时该改、何时不该改"，避免被频繁干扰也避免僵化。
**做法**: 识别三类合法触发器（按 §0 进化引子元规则）：
- **触发 A · 外部成熟体系借鉴**：发现某项目/工具已经在同类问题上有成熟设计（如 v1.0 → v1.1 被龙鱼五力 v3.2.1 触发）→ 通常是**轻调**（v1.x → v1.x+1），借鉴 4-6 项可直接套用的设计
- **触发 B · Doctor/专家反驳**：当评分/分析结果与产业链直觉矛盾时，Doctor 反驳的不是个案而是方法论缺陷（如 v1.1 → v1.1.1 被"二分法不公允"触发）→ 通常是**子方法论扩展**（v1.x.1 试点），不重写主方法论但增加场景化扩展
- **触发 C · 产业链事件触发**：当产业链发生重大事件（如 EUV 解禁、CPO 量产、Hi-ONE 进入商业化）→ 通常是**重构**（v1.x → v2.0），需要新增维度或重设权重
**反进化边界**（同等重要）：
- 一两个个案与直觉背离 → 个案处理（_meta 注记），不改方法论
- Doctor 临时性偏好 → 临时调整本次输出，不入方法论
- 短期市场情绪 → 不影响结构性评分
**为什么有效**: 显式化"进化触发器"让方法论的迭代节奏可控——既不僵化（合法触发就进化）也不浮躁（不合法触发就个案处理）。三类触发机制覆盖了方法论演化的所有合理来源。

### [EXP-20260609-003-P] 投研「叙事→事实」逐层降级核查法（成长股甄别）
**类别**: patterns
**场景**: 分析高估值成长股(尤其纪要/研报驱动)时,把"故事/指引"逐层逼到"已落地事实",防止把预期当现实定价。德科立688205实战(PE428×建立在未兑现的北美订单上)。
**做法**: 对每个利好主张,沿四个降级问连续追问,每降一级可信度递减——
1. **是谁说的?**(口径归属):公司自述/管理层指引 vs 专家测算 vs 媒体推算。"营收目标30亿"=公司guidance,非承诺非已实现;帕米尔纪要=专家转述公司说法,P1但仍是前瞻。
2. **是假设还是已落地?**(确定性):抓限定词"可能/有望/若…则/预计"=未发生;查在手订单/合同明细=已落地。德科立"下半年北美放量"全是"可能/若外溢",且在手订单不含北美→未落地。
3. **是单品/分项还是整体?**(口径范围):北美800G毛利37-38%(单品高值) vs 实际整体毛利25.7%(Tushare硬数据,被国内订单压住)。别用单品高值代表全盘。
4. **预期 vs 实际差多大?**(锚现实):目标30亿 vs 2025H1实际营收4.33亿;预期毛利40% vs 实际25.7%;故事净利+220% vs 实际ROE1.92%。差距越大,叙事成分越重。
**结果产出**: 区分"基准(敢写进目标)"vs"乐观上限(N个'如果'全中)"——德科立20亿=基准/30亿='踮脚够一下'(北美兑现+满产加班+外溢提前三条件齐发)。最后定位"市场在为哪一档定价":428×PE在按乐观上限+持续放量定价→预期差即博弈点/风险点。
**为什么有效**: 把模糊的"公司前景好"拆成可证伪的事实层级;每层都有硬抓手(口径归属/限定词/单品vs整体/实际财务),不被叙事带走。配 EXP-20260529-001(信源P级)用:前者判源可信度,本法判主张兑现度。
**来源**: → 德科立688205核查(帕米尔2026-06-02纪要 vs Tushare实跑) · 龙鱼-标的分析库 · logs/2026-06-09

### [EXP-20260609-004-P] 风险看议价结构而非集中度表象（单客户/集中/重资产的归因）
**类别**: patterns
**场景**: 评估"单一大客户/客户集中/重资产爬坡"等表象风险时,别被标签直接扣分,先判其议价结构与问题归属。源杰科技688498实战(单客户>53%曾被一刀切扣竞争维)。
**做法**: 对"集中/依赖"类风险,问四个结构问题再定扣分——
1. **有无自身替代者?**(被换掉的难易): 国产化率/技术梯队。源杰高端光芯片国产化率仅4%→难替代,是稀缺供给非易替代件。
2. **客户不采有无销路?**(需求广度): 看是否多买家。源杰实际覆盖中际旭创+新易盛+光迅+博创多家一线→销路不愁。
3. **溢价能力?**(定价权): 毛利率是硬证。源杰GM77.8%(持仓最高)→有定价权,非被压价弱势方。
4. **买方还是卖方市场?**(供需位势): 供不应求=卖方。AI驱动EML全球缺/Lumentum扩产40%→当前卖方。
**判定逻辑**: "稀缺供给端被龙头绑定"(强势·风险小) ≠ "易替代供应商依赖大客户"(脆弱·风险大),两者表象都是"单客户集中"但性质相反。**保留真风险**:源杰真威胁是①大客户自研一体化抽单 ②卖方市场周期性回摆,而非"集中"本身。
**为什么有效**: 把笼统的"集中度高=扣分"拆成可证伪的议价结构;高毛利+稀缺性+多买家=集中但不脆,反之才是真脆弱。配 ERR-20260609-001b(维度正交):风险也只扣其真实归属维。
**来源**: → 源杰科技688498核查 · 龙鱼-标的分析库 _meta.甄别_单客户风险性质 · logs/2026-06-09

### [EXP-20260610-001-S] 交付层 md 删除判据：DB+报告+已转html+无引用→删
**类别**: strategies
**场景**: 清理交付目录（如 AI4ME）冗余 md、决定哪些报告型 md 可删、哪些该移回项目层时。
**做法**: 四问全 yes 才删——①该项目有数据库/结构化数据支撑吗？②这份 md 是"报告"性质（非经验/方案/方法论）吗？③已转成对应 html 交付了吗？④全盘 grep 确认无任何引用（引擎/工作流/反链）吗？四 yes→可删（数据已在 DB/html）；任一 no→移回项目层保留。无 DB 的报告（如 PEC）、未转 html 的报告、经验/方案/方法论一律留。
**为什么有效**: 把"删不删"从直觉变成可证伪的四维判据，既清冗余又不误伤难复得的经验性内容。
