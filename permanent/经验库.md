# 经验库（agent 自我记忆）

> 与 [[通用教训]] 对称：通用教训记**错误与坑**，本库记**成功与可复用**。

```
**来源**: → [[项目]] 或 logs/日期-主题
```

### [EXP-20260521-001] 抖音单视频转字幕
**类别**: cases
**场景**: 拿到一个 `v.douyin.com` 短链或 `/video/` 直链，要它的字幕文本。
**做法**: Mac 终端 `cd ~/Documents/Claude/Projects/DVA && source dyd/.env.dva && node harvest-links.js "<链接>"`（链接加双引号）。沙盒/CC 环境**不能**跑（连不上抖音/volces/dashscope，已实测 DNS 解析失败）。
**为什么有效**: 走原始数据路径（下载 + ASR），不依赖 AI 多模态"间接读链接"，不被平台限流。
**来源**: → [[DVA]] · `DVA单视频最佳实践.md` · INFRA-20260521-001

## patterns · 可复用流程 / 方法

> 跨项目都能套用的工作范式。

### [EXP-20260523-001] 分级加载（L0/L1/L2）控制上下文
**类别**: patterns
**场景**: 记忆/文档多到一次性读会污染上下文、又怕漏关键信息时。
**做法**: 每个单元拆三层——L0 一句话摘要（判相关性）、L1 概览（做"要不要深入"决策）、L2 全文（确认深入才读）。brain 里 = stub abstract / 系统概览 / architecture 深文件。
**为什么有效**: 大部分判断只需 L0/L1，省 token 又不丢可达性；OpenViking 实测输入 token 降 83–96%。
**来源**: → [[CLAUDE]] 「分级加载约定」· brain-anchors skill

---

### [EXP-20260523-002] 修 error 会 unmask 同文件 warning
**类别**: patterns
**场景**: 用校验器（如 validate-frontmatter）批量修文件，修完一类问题后告警数反而"上涨"。
**做法**: 校验器常对同一文件既报 error（缺字段）又报 warn（值不合规），汇总按"有 error 即进 error 桶"。补完字段清掉 error 后，原被盖住的 warn 浮现、warn 数上升属正常。处理顺序：先拿 0 error 基线，再单独过 warn；别被"warn 变多"误判成"越修越糟"。
**为什么有效**: 区分"结构缺失(error)"与"值不合规(warn)"两级，warn 上浮是 unmask 而非新增。
**来源**: → logs/2026-05-23-brain校验清零与PEC中东油落盘

### [EXP-20260524-001] "revealed-preference 判别器"——把"X 是否吓退/奏效 Y"从情绪命题转成可观测判别器
**类别**: patterns
**场景**: 遇到"威胁/惩罚/政策是否吓退/奏效"类命题（如"天价罚单是否吓退外资""制裁是否逼退对手"），易陷情绪化断言（"肯定吓退"/"根本没用"）。
**做法**: 不直接下断言，而是**找当事人用脚投票的可观测行为**当判别器。① 先拆前提混淆（如"罚单吓退 Tesla"——Tesla 实际是关税僵局离场、根本没被罚，两套机制别混）；② 找最干净的 revealed-preference 信号（苹果一边在德里高院打 $38B 反垄断官司、一边扩 iPhone 印度产能 → 罚单威胁没吓退其制造 FDI）；③ 把命题写成 dated 观察项（"苹果不论 CCI 结果到 2027 底是否继续扩产"），用未来行为裁定。
**为什么有效**: 行为(revealed preference)比表态/情绪更硬；判别器把不可证伪的情绪命题钉成可观测的 dated 预测，直接对抗 G-01（橡皮图章）+ G-08（"有威胁→必奏效"未论证跳跃）。
**来源**: → [[PEC]] CS-07 A09 / predictions-register IN-P3 · logs/2026-05-24-PEC-印度制造业立国窗口与营商环境

## tools · 工具 / 接口 / SDK 心得

### [EXP-20260521-001-T] ASR 前先压缩音频
**类别**: tools
**场景**: 给 ASR（sensevoice-v1）准备音频、要走脆弱国内云链路上传时。
**做法**: 永远不传未压缩 WAV；先 ffmpeg 压成 AAC/m4a 16kHz 单声道（实测体积降到约 1/20）。异步任务（提交→轮询）每个网络调用都加重试。
**为什么有效**: 小文件在抖动链路上一次传完的概率大增；"缩小 payload"比"调网络"更治本。
**来源**: → [[DVA]] · INFRA-20260521-001

---

### [EXP-20260523-002-T] 重打包 Cowork .skill 包
**类别**: tools
**场景**: 改了 brain skill 的 SKILL.md，要把 `.skill` 安装包同步更新给 Doctor 重装。
**做法**: `.skill` = zip 包，布局为 `<name>/` + `<name>/SKILL.md`，压缩用 `ZIP_STORED`（对齐既有包）。沙盒挂载盘**禁 unlink**，覆盖既有包要"写临时文件 → `os.replace` 覆盖"（同目录 rename，允许）。打完用 `zipfile.testzip()` + 列 `namelist()` 验证。装包入口可用 present_files（.skill 自带 install 按钮）。
**为什么有效**: 匹配安装器预期布局；os.replace 绕开挂载盘 unlink 限制；先验证再交付不翻车。
**来源**: → logs/2026-05-23-记忆系统对标OpenViking与五项优化落地 · [[通用教训]]（沙盒只 rename 不 unlink）

### [EXP-20260523-003-T] 沙盒只读 git 姿势（不留 index.lock）
**类别**: tools
**场景**: 沙盒里要看 git 状态/历史，但 FUSE 挂载盘连只读 `git status` 都会留 `.git/index.lock` 且删不掉。
**做法**: 一律加 `--no-optional-locks`（如 `git --no-optional-locks log --oneline -5` / `status --porcelain`），它不刷新索引、不建锁。判断"自己刚改的文件是否已被他会话卷走"用 `git show HEAD:<path>` 比对磁盘内容；探测 .git 存在性优先用 Read/ls 而非 git。任何写操作贴命令给 Doctor（G-X2）。
**为什么有效**: `--no-optional-locks` 跳过"刷新索引"这一步——而正是刷新索引在尝试建 index.lock；`git show HEAD:` 直读对象库不碰工作区/索引。
**来源**: → [[通用教训]] G-X2补 / G-X9 · logs/2026-05-23-brain校验清零与PEC中东油落盘

## strategies · 打法 / 策略

> 解题策略、协作打法、元方法。

### [EXP-20260523-001-S] 对标外部方案先分"借/拒"，按自身规模与模型裁剪
**类别**: strategies
**场景**: 看到一个"更先进"的外部系统（如 OpenViking），想借鉴来改自己的系统。
**做法**: 先摸清自己系统的**真实**架构（读源文件不靠假设）→ 逐维度优劣对比 → 按**自身规模 + 协作模型**判断哪些值得借、哪些是过度工程，明确列出"不做"项及理由 → 分级提案待批。
**为什么有效**: 外部方案的设计前提（如自主 agent / 海量记忆）未必是你的前提；机械照搬会引入不匹配的复杂度。"明确不做"和"做"一样有价值。
**来源**: → [[OpenViking-Agent记忆系统]] · [[架构决策-20260514]] D6

---

### [EXP-20260523-002-S] 元数据批量修复"dry-run→批准→apply"范式
**类别**: strategies
**场景**: 要批量改一批文件的 frontmatter / 元数据（补字段、归一取值），既要快又怕改错或破坏既有内容。
**做法**: 一套逻辑写两遍——先写**只读 dry-run 脚本**逐文件算出拟改值并打印（旧→新），交 Doctor 过目；批准后把同一取值逻辑改成 **apply 脚本**落盘；最后**重跑校验器**验证清零。沙盒写失败的个别文件（FUSE 权限）用 Edit 工具兜底。取值规则透明化（日期取文件名/git、title 取 H1、tags/type 按路径推断）便于人工核。
**为什么有效**: dry-run 与 apply 共用同一函数 → Doctor 看到的就是将落盘的；只读预演把"批准"前置，把不可逆写操作关进人在环。
**来源**: → logs/2026-05-23-brain校验清零与PEC中东油落盘 · [[通用教训]]（G-X8 已落盘假象）

### [EXP-20260524-001-S] 闭环/万能命题遇反例 → 划"适用线域"，不推翻也不无视
**类别**: strategies
**场景**: 一个已沉淀的解释性闭环/强命题（如 CS-07 印度"现实失败→叙事胜利→继续投入"）遇到硬数据反例（iPhone 印度产能占全球约 1/4 = 真实物质捕获，不是"叙事胜利"）。
**做法**: 三选项里选第三条路——① 不强行把反例塞进闭环解释（那会让闭环能解释一切 = G-01 橡皮图章）；② 不因一个反例就推翻闭环（其原线域证据仍强）；③ **划定闭环的适用线域**：把"全域机制"降为"某线域机制"（闭环收窄为军事-外交线机制 1962/Sindoor，经济线存在真实捕获 = 部分证伪 + 命题收窄）。落盘时显式记"部分证伪 + 边界"，并 promote 回写源命题。
**为什么有效**: 既保住命题在有效线域的解释力、又诚实承认反例，是 G-10（框架自我保护）的**建设性防御**——框架靠"划边界"进化，而非靠"打补丁"续命（对接 predictions-register 更正协议的进步/退化判据）。
**来源**: → [[PEC]] 决策记录 2026-05-24 闭环分线边界裁定 · CS-07 A09 · logs/2026-05-24-PEC-印度制造业立国窗口与营商环境
