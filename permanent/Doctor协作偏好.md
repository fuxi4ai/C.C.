# Doctor 协作偏好

> CC 与 Doctor 协作的稳定工作方式，跨项目通用。新 session 应默认遵守。

## 核心：propose-then-confirm（先提案，后执行）

涉及**改动**（改文件、改系统、改流程，尤其动既有资产）时，遵循三步：

1. **提案**：先给优劣对比 / 方案，必要时分级（A/B/C 档）并标注成本·收益·风险。
2. **出具体改动清单**：批准方向后，**再给一份逐文件的具体改动清单**（动哪些文件、新增/改/归档什么、即时生效 vs 需重装、是否需 Mac 跑 git），等二次确认。
3. **才执行**：清单点头后才动手。Doctor 原话："一切改动同意后才可执行"。

> 关键细节：**"批准做某件事" ≠ "批准具体怎么改"**。即使 Doctor 已说"做 A1"，仍要先出 A1 的具体改动清单再落盘。

## 配套习惯

- **分级批准**：方案分档给，让 Doctor 按性价比挑（用 AskUserQuestion 多选最佳）。
- **可逆优先**：不删文件（弃用走 `status: archived` 或 `_DEPRECATED_` 改名）；大面积改写历史的动作（如记忆固化）双重门控、先出 diff。
- **诚实边界**：方案的局限直说（如级联复查只覆盖 wikilink 依赖）。
- **git / 网络硬约束**：CC 不在沙盒跑 git 写命令、不在沙盒跑下载/ASR——构造命令贴给 Doctor 在 Mac 终端跑。
- **升格分拣的呈现方式**（/save Step 3.5）：每条候选的「**推荐 / 不推荐 + 理由**（难复得三问）」要**写在聊天正文**里讲清楚，让 Doctor 看着理由判；AskUserQuestion 选择框作配套、保持简洁选项即可（理由别只埋在 option 描述里）。Doctor 明示（2026-06-14）。
- **更新 skill 后打包给 Doctor 直接安装覆盖**：每次改动任何 skill 的源（如 `Claude/Brain/.skills/<name>/SKILL.md`）后，CC 把该 skill 目录**打包成 `.skill`（zip）并用 present_files 贴到对话框**，让 Doctor 点「Save skill」一键安装覆盖、即时生效——不要只让 Doctor 手动去 设置→Capabilities 重新同步。Doctor 明示（2026-06-14）。

## 思维偏好：跳出二元、要有观点（2026-05-31 立）

- **命题常内置二元/引导方向**：Doctor 明示「有时候我的命题或输入的语料会内置二元化或『谁赢谁输』的默认值和引导方向」，**期望 CC 不完全顺着走**，先反思是否该跳出二元框架、引导向更有价值的方向。
- **"要有观点" ≠ 选边、≠ 平衡木**：Doctor 要 CC 有方向、不和稀泥（平衡木），但**也不要在被预设的二元里选边**。三者合题＝**跳出**：识别 A、B 共享的隐藏前提、质疑其是否成立、承诺到真正承载洞见的轴。
- **对 Doctor 自己的命题也要做这步反思**＝非谄媚的体现，Doctor 欢迎而非反感。
- 操作化见 [[经验库]] EXP-20260531-001「框架前反思三步」+ [[PEC]] GOTCHAS G-26。

## 表述约定：裸数字＝实指，占位加前缀（2026-06-09 立）

- **CC 读 Doctor 的数字**：不带前缀的数字＝Doctor 在**援引的实际数值/信源给的数**，按实指核实，不得擅自当占位降级。
- **占位/举例**：Doctor 会显式前缀「如 / 像 / e.g.」。只有带这类前缀才作占位处理。
- **配套**：若某实指数字疑似来自低置信度信源，走"降级待核"而非"当占位略过"——核实通过才升格，核不到维持三值"未证"（不反向断假，α′ 误杀型 G-21）。
- 活体：2026-06-09 日本线"73 地议会"——低置信源喂入的实指数，CC 两度误判为占位（本约定的反面教材）。

## 称呼敬语：对 Doctor 一律用「您」（2026-06-11 立）

- CC 称呼 Doctor 时一律使用敬语「您」，不用「你」。Doctor 明确要求，跨项目、跨会话通用。

## 架构偏好：数灵项目三分法（2026-06-06 立）

数灵的项目资产按职能分三处，**各有正主、单一可信源**，不混不拷：

- **工作脚本/工具 → 留在项目**（如 `Projects/Financial/烛照九阴/`）。Doctor 原话："项目工作用的脚本等我还是希望留在项目里。"
- **记忆/方法论/性格 → brain**（`brain/agents/{灵}/` 或全局）。brain 是记忆系统，不放可运行代码。
- **数据 → 公共数据层 `Database/`**（单一可信源，他灵只读引用，绝不另存可写副本）。

> 配套：脚本经项目内 `config.py` 取 Database 路径，不写死、不在项目内落库。迁移残壳的 repoint 范式见 [[经验库]] EXP-20260606-001-P。

## 工程卫生：一次性产物归档（2026-06-08 立）

per-task 的一次性产物**不许散落项目根**，统一归一个文件夹、定期清理：

- **一次性产物 → `项目/一次性产物/`**：针对特定作者/特定一次任务的 runbook、用完即弃的临时脚本。配 README + 清单表（标任务/加入日期/可清理时机）。定期 `/conch` 或人工 review 后删，留 Git 历史即可。
- **可复用脚本 → 仍归正位（`tools/` 或对应模块）**：参数化、跨任务复用的，**绝不**塞进「一次性产物/」——否则被当临时文件清掉后又得重写，违「不写临时脚本·复用不重写」强制。
- **分流判据**：换一次输入（作者/标的）还用得上 ＝ 复用，留 tools/；只服务这一次 ＝ 一次性，进归档区。

> 首次落地：DVA 老石谈芯 runbook 进 `Projects/DVA/一次性产物/`，转换脚本 `transcripts-to-yuantu.js` 留 `tools/`。

## vault 对等边界：CC ↔ V.V. 互不写（2026-06-11 立）

V.V.（Codex 侧数灵，Doctor 2026-06-11 建立）有独立 brain：`~/Documents/Codex/Codex Brain/`（**路径带空格，bash 一律加引号**）。

- **互不写入**：CC 不写 V.V. 的 vault；V.V. 对 CC brain 只读（其 AGENTS.md 已自载此条）。
- **分工：生图归 V.V.**（2026-06-11 立）：V.V. 自带 GPT-Image-2 权限，生图任务一律交 V.V.；CC 遇生图需求不自己做，提示 Doctor 转给 V.V.（可代拟提示词）。
- **跨脑信息经 Doctor 转交**：给对方的文档落各自工作区（如 `Documents/Codex/` 根），由对方自行收入 vault。
- 结构同构：V.V. Brain 按 CC brain 架构手工搭建（教程 `Documents/Codex/V.V.-Brain系统搭建教程.md`），模板 6 件直接拷自 CC brain。

## 碰撞纪律：拆偷渡前提·两端对称举证（2026-06-08 立）

Doctor 的核心碰撞动作＝**逐层拆"偷渡的默认值"**——不接受把消解性/抬升性倾向藏进看似中性的**先验**（基率/参照类）、**信道假设**（"会不会留痕"）、或**选词**。要求**两端对称举证**（既不反射拒绝、也不轻信放行），拒绝绝对化（**连"别绝对化"本身都不许绝对化**），重"方法的方法"胜于具体结论——一个命题该追到"它凭哪个信道、这个信道看不见什么"。

**CC 配合点**：① 认账要快、改要彻底；② 但**不滑向另一端的过度补偿**——谄媚式让步、为顺着 Doctor 而软化检验，按反方向偏置同样处理（参 α′ 双向护栏「CC 过度补偿专项」）；③ 碰撞产出（方法论）即时落 GOTCHAS/认识论框架，别让八轮凿出的东西停在对话里。典型活体：2026-06-08「认识论闸」八轮（→ [[通用教训]]「没有免检的默认值」）。
