## 存储与机制（关键，先懂这个再迁移）

- **任务文件**：`~/Documents/Claude/Scheduled/{taskId}/SKILL.md`（受保护目录，沙箱挂不了，只能 Doctor 终端 cat 出来 / 用 `create_scheduled_task` 写）。
- **`SKILL.md` 的 frontmatter 只有 `name` + `description`**——**调度信息（cron/每天几点）不在文件里**，写在 description 的散文里给人看。真正的 `cronExpression` / `enabled` / `nextRunAt` 存在一张**独立的注册表**。
- ⚠️ **故"光有 SKILL.md 文件 ≠ 任务已注册"**：切环境后旧文件还在 `Scheduled/` 里，但新环境注册表是空的 → `list` 返回 0。**必须用 `create_scheduled_task` 重建**（它把 prompt 正文 + cron 一起登记），不能只靠拷文件。
- **jitterSeconds（确定性延迟）**：平台在调度时刻给每个任务加几分钟随机偏移分散负载（如设 06:15 实跑 06:20、06:30→06:38）。**正常**，只要任务间相对顺序不乱即可。
- **cron 用 user 本地时区（中国 UTC+8）**，不是 UTC，直接写本地时间。
- **cron 的 `,` 列表（如 `3,6`）有效**，但人类可读摘要可能渲染不全（实测 `30 8 * * 3,6` 摘要显示「only on Wednesday」漏了周六，**底层 cron 是对的、照常周三+周六跑**）——以 `list_scheduled_tasks` 返回的 `cronExpression` 字段为准，别信摘要那行。

**⏰ 时区 + 中立语境（2026-07-01 定）**：cron 走**机器本地时区（美西 PDT/UTC-7）**；数据逻辑按**"前一交易日" + 交易日历**判断，**不绑钟点**。目标市场在 **UTC+8** 时区，故这些美西本地时刻大致落在目标时区的清晨窗口（`14:45美西→次日05:45 UTC+8`…`17:30→08:30`）。**prompt 文案统一用「中立语境」**——去掉一切钟点/时段措辞（"06:xx"、开盘前、静默时段、早安等），只留数据逻辑；**且不在文案里显式标注地域**（合规/账号安全考量）。好处：再挪时间不必改文案。

**依赖链顺序**（美西本地 · jitter 后相对顺序仍保持）：行情 14:45 → 入库 15:00 → 审核 15:30 → 日报 16:00 → 看板 17:30。
- 审核(15:30) 在 入库(15:00) 之后 ✓ · 日报(16:00) 依赖 market-data(14:45) 在其后 ✓ · 看板(17:30) 在 market+zhuzhao 之后 ✓ · 入库↔行情互不依赖、调换无碍 ✓
- 投知君君 周三/六 17:00 美西 独立、无下游依赖。
- baize 周报 周日 01:00 美西（原 14:30，"留足本地 cron 缓冲"旧理由已随 Mac cron 退役失效）。

**⚠️ 待核·星期漂移（美西 cron → UTC+8 目标时区 +1 天）**：weekday 任务是 `* * 1-5`（美西周一~五）→ 落到目标时区**周二~周六**（美西 Fri→目标 Sat〔休市、空跑无害〕；**美西周日无 cron → 目标时区周一清晨无专班**）。若要目标每个交易日清晨都有班，cron 星期域可能需纳入周日。标此待核。

**通知策略（2026-06-30 Doctor 定）**：日常晨间静默类（行情/课件/看板）不通知；只有**周报 / 月报 / 记忆体检**这类成果报告通知。

## 网络白名单：gateway 不需要（开放出网）

切环境时 Doctor 关心的一个点——官方工作区这些任务能取数，靠把 `api.tushare.pro`/`hq.smm.cn`/`api.deepseek.com`/`stockanalysis.com`/Yahoo/抖音 加进沙箱**网络白名单**。**gateway/Claude-3p 不需要**：

- **官方应用**：默认拒绝 + 逐域名加白（要一个个申请放行）。
- **gateway/Claude-3p**：**开放出网**——2026-06-30 实测连官方从没加白的 `github.com`/`example.org`/`baidu.com`/`httpbin.org` 都直接 200，无 allowlist 闸。这些任务的取数域名**开箱即通，无需任何白名单配置**。
- **诚实边界**：实测的是交互式 workspace 沙箱；定时任务在**调度时刻**跑的沙箱理论上可能策略不同。最稳验证＝对某取数任务点一次 **「Run now」**，既验真实出网、又**预批工具权限**（bash/web_fetch/update_artifact），免得定时跑时卡权限弹窗。

## 路径解析：gateway 平铺挂载 → 脚本 env 覆盖（G-X45 · 两批全完 2026-07-01）

gateway 把请求挂载的目录平铺到 `mnt/{basename}`，不还原 macOS 层级——脚本靠 `parents[N]`/向上找 `Documents` 定位兄弟目录（输出区/库/AI4ME）会算错。解法：脚本加 env 后门（有 env 用 env、无则回退老逻辑，Mac 原生零影响），任务 prompt 补「前置：挂载 + 路径 env」块导出之。

| 任务 | 脚本 | env 变量 |
|------|------|----------|
| zhuzhao-…-daily-report | `gen_daily_report.py` / `market_health.py` | `ZZJY_OUTPUT_ROOT` · `ZZJY_ARTIFACT_ROOT` · `ZZJY_DATABASE_ROOT` · `MARKET_DATA_DIR` |
| baize-weekly-report | `build_weekly_report.py` · `sync_to_cowork_artifact.py` · `weekly_health.py`（+ `lib_public_read`/`lib_paths`） | `BAIZE_OUTPUT_ROOT`(AI4ME) · `BAIZE_DATABASE_ROOT`(Database·龙鱼库) · `BAIZE_ARTIFACT_ROOT`(Artifacts)；`BAIZE_BUSINESS_DB` 更具体优先 |
| refresh-asset-dashboard | `conch_engine.py` | `CONCH_DOCUMENTS_ROOT`(survey 解析各项目相对路径的 Documents 根) |

第一批（烛照/行情线）2026-06-30、第二批（白泽/海螺周更线）2026-07-01。详见 [[通用教训]] G-X45。

## 切环境重建 SOP（下次照做）

1. 新环境 `list_scheduled_tasks` 若返回空 → 旧任务没带过来。
2. Doctor 终端导出旧任务全文：`find ~/Documents/Claude/Scheduled -name SKILL.md | while read f; do echo "### $f"; cat "$f"; done`（或直接照本清单的 cron + 各 SKILL.md）。
3. 逐个 `create_scheduled_task`（taskId / cronExpression / description / 完整 prompt 正文 / notifyOnCompletion）；ad-hoc 任务省略 cron。
4. `list_scheduled_tasks` 核对 `cronExpression` 字段（别信摘要行）。
5. 挑一个取数任务「Run now」验真实出网 + 预批工具权限。
6. prompt 正文一字不差照搬旧 SKILL.md（这些任务规约极细，漏字会跑偏）。
